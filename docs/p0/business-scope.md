# 阶段 0 · 私募业务场景说明 v0.3

> 文档状态：阶段 0 业务边界草案  
> 适用业务：私募内部投研与组合决策；FOF / 母基金作为可配置扩展模式  
> 阶段 0 目标：冻结真实业务入口、候选对象、证据要求和最小决策状态，不写新业务代码。

## 1. 业务结论

当前系统不应默认做面向大众的基金推荐，也不应在未确认业务模式前锁定为 FOF。

阶段 0 的默认业务模式是：

```text
私募内部研究请求
→ 投资假设
→ 行业、股票、基金或策略候选
→ 估值与风险证据
→ 组合提案
→ 最小决策状态
```

如果公司确认主要做 FOF / 母基金，则启用 `fof` 模式：

```text
外部基金/管理人研究请求
→ 管理人和基金尽调
→ 持仓与策略穿透
→ 候选集合
→ FOF 组合提案
```

同一套系统可以支持两种模式，但阶段 0 只选择一个主模式跑通，避免同时建设两套完整产品。

## 2. 研究请求而不是外部用户

业务层统一称为“研究请求”，技术层暂时保留 `user_input_id`，以降低后续迁移成本。

每个研究请求必须包含：

- `actor_role`: researcher / portfolio_manager / risk / product
- `request_source`: research_meeting / ad_hoc_research / portfolio_review / risk_review
- `business_mode`: private_strategy / fof
- `strategy_policy_id`
- `raw_text`
- `data_snapshot_id`
- `created_at`

不可接受的流程：

```text
系统自行运行
→ 自动产生候选
→ 没有发起人、策略和数据上下文
```

必须采用：

```text
研究请求
→ 可复现的分析
→ 候选和排除原因
```

## 3. 三个研究请求场景

### 场景 A：我有投资观点

示例：

- “我认为 AI 将持续提升生产力。”
- “我认为创新药被市场错杀。”
- “我偏好低估值、高现金流和稳定分红。”
- “我认为当前行业景气度已经过高，不想追高。”

输入内容：

```text
raw_text
business_mode
strategy_policy_id
time_horizon
conviction
```

系统处理：

```text
研究请求
→ InvestmentThesis
→ 结构化观点
→ 支持/反向证据
→ 行业、股票、基金或策略候选
→ 估值和风险检查
```

private_strategy 模式的候选对象：

- 股票。
- 行业。
- 产业链环节。
- 自有策略。
- 自有产品。
- 外部基金作为竞品或工具。

fof 模式的候选对象：

- 外部基金。
- 外部管理人。
- 管理人旗下代表产品。

验收要求：

- 有证据的候选可以进入候选集合。
- 没有合格候选时，返回 `no_eligible_candidate`。
- 每个候选必须有匹配原因、数据日期、估值状态和排除原因。
- 不要求固定数量，不允许为了达到数量强行补候选。

### 场景 B：我有看好的行业或方向

示例：

- “我看好 AI 产业链。”
- “创新药可能进入新周期。”
- “高股息资产适合当前防守。”
- “新能源出口可能超预期。”

系统处理：

```text
行业/方向
→ 行业和产业链拆解
→ 股票和公司映射
→ 持仓穿透
→ 估值和预期差
→ 策略适配
→ 候选集合
```

必须展示：

- 行业和产业链层级。
- 相关股票和公司。
- 行业暴露或持仓权重。
- 数据报告期。
- 估值状态。
- 支持和反向证据。
- 当前策略是否允许该方向。
- 候选被排除的原因。

FOF 模式下，再将股票和行业暴露映射到外部基金；private_strategy 模式下，优先输出股票、行业和策略候选。

验收要求：

- 输入一个行业后，能返回真实穿透结果或明确数据不足。
- 不以“至少三组基金”作为成功条件。
- 没有可用暴露时，必须解释是没有持仓、数据缺失还是策略排除。

### 场景 C：我知道具体标的

输入类型：

```text
stock
fund
manager
strategy
product
```

隐含研究意图：

```text
copy
alternative
hedge
correlate
understand_exposure
```

股票输入时：

- 查询真实持仓该股票的基金。
- 查看上下游和同产业链股票。
- 分析现有组合是否已有相似暴露。
- 找到替代、对冲或互补标的。
- private_strategy 模式下，支持进入股票候选池。
- fof 模式下，支持进入基金候选池。

基金输入时：

- 解析基金真实持仓。
- 计算基金之间的持仓重叠和风格差异。
- 区分同管理人产品和不同管理人相似产品。
- 输出可替代、可互补和高重复候选。

管理人输入时：

- 展示管理人旗下产品。
- 分析策略风格、经理稳定性和产品重叠。
- FOF 模式下进入管理人尽调。
- private_strategy 模式下作为外部竞品分析，不自动进入自有策略。

验收要求：

- 能返回真实持仓、报告期和数据质量。
- 能说明候选是复制、替代、对冲还是重复暴露。
- 数据不足时返回门禁原因，不返回虚假的完整候选。

## 4. 内部角色和决策边界

| 角色 | 主要输入 | 主要输出 |
|---|---|---|
| 研究员 | 研究请求、数据和历史案例 | InvestmentThesis、CandidateSet |
| 投资经理 | 研究假设、候选、组合影响 | 组合提案、最小决策状态 |
| 投决会成员 | 投决材料、风险检查、反向证据 | approved / rejected / watching |
| 风险管理 | 策略政策、组合暴露、监控事件 | 风险提示、挂起或升级 |
| 产品/运营 | 策略政策、产品目录、数据批次 | policy version、数据快照 |
| 系统管理员 | 权限、运行和数据源 | 仅阶段 5 纳入 |

“用户”只作为技术兼容字段和请求发起人的统称，不作为外部客户角色。客户适当性、销售和财富管理不在本 P0 范围。

## 5. 阶段 0 范围

### 必须完成

- 冻结 private_strategy 和 fof 两种业务模式的字段边界。
- 提供一份自有权益策略 Policy 样例。
- 保留一份 FOF Policy 样例作为可选扩展。
- 跑通 A/B/C 三类研究请求的结构化演示。
- 每个结果绑定策略版本和数据快照。
- 每个候选有证据、数据日期和排除原因。
- 允许无候选状态。
- 明确阶段 1 的最小治理对象。

### 不在范围

- 完整自然语言理解。
- 自动生成投资观点。
- 自动下单或券商连接。
- 客户适当性和财富管理。
- 跨策略统一收益排名。
- 实时盘中监控。
- 多产品组合聚合。
- 完整投决会权限和审批。
- 真实的预测收益承诺。

阶段 0 可以使用固定结构化样例，但必须保留原始文本，不得把样例伪装成完整自然语言能力。

## 6. 阶段 0 复现要求

每个演示结果必须能够关联：

```text
research_input_id
→ thesis_id
→ candidate_set_id
→ strategy_policy_id + version
→ data_snapshot_id
```

复现报告必须能够回答：

- 谁发起了研究请求。
- 请求原文是什么。
- 使用了哪个策略政策。
- 使用了哪批数据。
- 哪些候选被保留。
- 哪些候选被排除。
- 是否存在数据不足。
- 当前结果是研究候选还是组合提案。

## 7. 阶段 0 出口

阶段 0 只有在以下条件全部满足后，才能进入阶段 1：

- 主业务模式已由管理层确认。
- 私募内部投研和 FOF 扩展的边界已经写清楚。
- 领域字段命名没有 `strategy_id` / `policy_id` 等歧义。
- 至少一份主业务策略政策样例通过投研负责人评审。
- A/B/C 三类请求都有一份可复现报告。
- 可以合法返回无候选结果。
- 快照架构和复现命令已明确。
- 阶段 1 的最小决策记录范围已确认。

## 8. 阶段 1 入口

阶段 1 默认实现：

```text
ResearchInput 落库
→ InvestmentThesis 落库
→ CandidateSet 落库
→ DecisionRecord 最小版
→ /inputs 研究请求入口
→ /inputs/:id 证据和候选响应页
```

FOF 模式只有在公司确认外部基金筛选是实际业务后，才增加管理人尽调和外部基金准入字段。
