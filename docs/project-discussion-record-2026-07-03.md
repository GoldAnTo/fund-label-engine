## 一、项目整体总结

### 1.1 项目定位

`fund-label-engine` 是一个基金标签计算引擎，核心目标是把基金数据加工成可解释、可追溯、可复核的标签结果。项目分两个层次完成了闭环：

**第一层：标签和证据闭环**

- 基金范围：Phase1 v1 正式清单 142 只权益相关基金（股票型、混合型-偏股、混合型-灵活、指数型-股票）
- 数据 pipeline：从 fundData SQLite 读取基金画像、NAV、持仓、行业、经理、费率，外挂股票因子库做持仓穿透
- 8 步计算流程：加载材料 → 确认范围 → 数据 gate → 基础指标 → 持仓风格 → 相对基准 → 打标签 → 分类分组
- 基准精确解析：逐只解析业绩比较基准，拆组件、合成日收益，不用错误基准代理
- 当前结果：142 只全部 `label_ready`，其中 114 只 `relative_label_ready`（可展示 Alpha/Beta/超额收益），28 只暂不可展示
- 标签体系：覆盖数据质量、收益风险、持仓结构、基金经理、费用规模、风格边界、高级风格、相对基准等类别
- 计算状态：每个标签记录 `triggered` / `not_triggered` / `not_computed`，解释为什么出/没出
- 工程化：Makefile 一键跑批、CI 覆盖 pytest + 前端 build + smoke test、FastAPI API、React 工作台

**第二层：组合草案闭环（6 月 30 日后新增）**

- 组合角色推导：基于标签自动推导核心候选、卫星阿尔法、防守锚、指数工具、低成本等角色
- 风险约束：按风险标签（高回撤 3%、高波动 5% 等）收紧权重上限，硬阻断标签直接排除
- 组合草案引擎：评分排序 → 排除硬阻断 → 按核心/卫星/指数工具桶分配 → 应用上限 → 重新分配
- 人工校准：`portfolio_role_reviews` 表 + API + 前端工作台，研究员可签署 accept/reject/needs_more_data
- 当前结果：28 只 eligible（9 core + 11 satellite + 8 index_tool），88 只进入草案，54 只被排除

### 1.2 基准缺口处理

- 已修复：第一批 5 只精确指数代码确认（+5 ready），第二批 `000172` 固定年化合成（+1 ready），ready 池从 108 提升到 114
- 已清零：`benchmark_mapping_required` 和 `benchmark_unresolved` 均为 0
- 剩余 28 只：21 只债券总类需授权源、6 只供应商/主题/策略指数需授权源、1 只未披露基准需人工确认

## 二、从基金投研机构视角看项目价值

### 2.1 当前定位

当前项目核心是"产品池准入和标签证据系统"，使用方更接近：

- FOF 基金经理 / 基金中基金管理人
- 银行/券商的基金研究团队
- 第三方基金评价/筛选平台
- 财富管理平台的基金准入审核

对基金公司自身（管理基金的一方）也有用，但用途和优先级不同。基金公司最关心的是：

```text
我的基金在同行里排第几？
我的基金经理有没有跑赢同类基准？
市场上什么风格稀缺/拥挤，我该发什么产品？
销售时怎么用标准化标签讲清楚基金画像？
```

### 2.2 核心价值评估

当前项目最大的资产不是那 142 只基金的标签结果，而是它建立起来的方法论和工程框架：

```text
数据 gate → 指标计算 → 规则标签 → 证据落库 → 计算状态 → 分类分组 → 基准门禁 → 组合草案
```

这套框架是可扩展的。把基金范围从 142 只扩到 5000 只，引擎本身不用改，只需要数据接入和跑批资源。把标签从"是否触发阈值"升级到"同类分位数"，也是加字段和改规则，不需要重构。

对基金公司来说，这个项目的价值在于：它提供了一个把基金研究流程标准化、可审计、可复现的基础设施。

### 2.3 价值矩阵

| 价值点 | 当前完成度 | 对基金公司的重要性 |
|---|---|---|
| 标签标准化和证据链 | ★★★★★ | 高——解决了"标签口径不统一"的老问题 |
| 基准精确解析 | ★★★★★ | 高——这是很多机构做错的地方 |
| 相对基准标签门禁 | ★★★★☆ | 高——避免用错误基准误导结论 |
| 组合角色和权重草案 | ★★★★☆ | 中——对 FOF 有用，对主动管理公司用处小 |
| 风格识别 | ★★★☆☆ | 中——有雏形但样本少、单期 |
| 风险监控 | ★★★☆☆ | 中——有标签但没有时序和预警 |
| 竞品横评 | ★★☆☆☆ | 高——但样本太少且缺排名 |
| 基金经理考核 | ★★☆☆☆ | 高——但缺多周期和业绩拆分 |
| 产品策略 | ★★☆☆☆ | 中——样本不够代表市场 |

## 三、实际使用场景

### 场景 1：竞品横评——把自家基金和同行放在同一把尺子下

当前能力：系统已经能对 142 只基金用统一规则算收益、风险、风格、费率、相对基准标签，每只基金都有证据链。

基金公司怎么用：投研团队选一个风格（比如"质量成长"），系统列出所有带 `quality_growth` 标签的基金，包含自家和竞品。研究员可以横向比较 Alpha、夏普、回撤、费率，看自家基金在同类里位置。

当前缺口：
- 142 只样本太少，覆盖不了全市场同类基金
- 缺少分位数/排名维度（当前只有"是否触发阈值"，没有"在同类里排前 20%"）
- 缺少按基金公司聚合的视图（看自家整体 vs 同业整体）

### 场景 2：基金经理考核——用相对基准替代绝对收益

当前能力：系统已经实现了基准逐只解析、Alpha/Beta/信息比率计算，且有 `relative_label_ready` 质量门禁，不会用错误基准凑数。

基金公司怎么用：考核基金经理时，不看绝对收益排名（牛市谁都能涨），而是看：
- `excess_return_strong`：有没有跑赢自己的基准
- `information_ratio_high`：跑赢的效率高不高
- `alpha_positive`：扣掉基准后还有没有正贡献
- `tracking_error_high`：偏离基准是不是太大

这比传统"年度收益率排名"更公平。

当前缺口：
- 缺少多周期对比（当前只有 1Y/3Y，基金经理考核通常要看 1Y/3Y/5Y/任职以来）
- 缺少经理变更后的业绩拆分（当前只看在任经理任期，没有把前任和现任业绩切开）
- 缺少风格漂移的正式展示（当前 `style_drift` 只是观察标签）

### 场景 3：产品策略——市场上什么风格稀缺/拥挤

当前能力：系统已经能识别深度价值、质量成长、红利稳健、金融高股息、消费质量等风格标签，并统计每类风格有多少只基金。

基金公司怎么用：产品部看风格分布：
- `quality_growth` 只有 9 只 → 这个风格在市场上还稀缺，可以布局
- `industry_concentration_high` 有 86 只 → 行业集中型基金太多，同质化严重
- 带费率较低标签的有多少 → 低费率指数工具的竞争格局

当前缺口：
- 142 只样本不代表全市场供给
- 缺少风格 × 规模交叉（稀缺但都是迷你基金，和稀缺且百亿规模，含义不同）
- 缺少资金流向维度（什么风格在净流入）

### 场景 4：风控监控——自家基金的风险标签变化

当前能力：系统已实现 `drawdown_high`、`volatility_high`、`beta_high`、`holding_concentration_high`、`industry_concentration_high`、`tracking_error_high` 等风险标签，并在组合层用这些标签收紧权重上限。

基金公司怎么用：风控团队每月跑批，看自家基金的风险标签是否从 `not_triggered` 变成 `triggered`。比如某只基金上个月还是行业分散，这个月突然 `industry_concentration_high` 触发了，说明基金经理可能调仓集中了。

当前缺口：
- 当前是静态快照，没有标签时序变化（run diff 能对比两次 run，但没有标签级别的时间序列）
- 缺少预警机制（标签状态变化时主动推送）
- 持仓数据是季频，有滞后性

### 场景 5：销售支持——标准化基金画像

当前能力：系统已经把标签分成正式结论、观察信号、待校准信号、阻断原因四层，每个标签都有证据。

基金公司怎么用：销售团队拿到一份标准化基金画像：
- 正式结论：数据充足、长期收益优秀、夏普较高、Alpha 为正、经理任期长
- 观察信号：行业集中度较高
- 待校准：质量成长风格（样本少）

销售面对客户时，用这套标签快速讲清楚基金特征，而不是翻几十页研报。

当前缺口：
- 缺少面向客户的展示层（当前工作台面向内部投研，不是面向客户）
- 缺少基金画像 PDF/海报导出
- 缺少同类排名和历史得奖信息

### 场景 6：FOF 子基金筛选——核心-卫星组合构建

当前能力：组合工作台已经实现角色推导、风险约束、权重上限、组合草案、人工校准。

基金公司怎么用：FOF 基金经理用组合工作台筛子基金：
- 核心池：9 只，权重上限 6-8%
- 卫星池：11 只，权重上限 3-5%
- 指数工具池：8 只，权重上限 3%
- 风险标签自动收紧上限
- 硬阻断标签自动排除

这比手动 Excel 筛子基金高效得多。

当前缺口：
- 142 只子基金池太小，FOF 通常需要覆盖几百上千只
- 缺少相关性矩阵（核心池里几只基金之间相关性高不高）
- 缺少行业/风格暴露约束（组合层没有做行业偏离约束）
- 缺少申赎成本和流动性约束

## 四、142 只基金限定下的最终目标：六个模块

### 4.1 数据底子

| 数据维度 | 当前情况 | 能支撑的分析 |
|---|---|---|
| NAV 日收益 | 142 只都有 1Y，3 只有 3Y | 收益、波动、回撤、夏普、相对基准 |
| 股票持仓 | 最新一期，含权重 | 持仓集中度、权益仓位、风格穿透 |
| 行业配置 | 最新一期 | 行业集中、行业分散 |
| 基金经理任期 | 有任期天数 | 经理稳定性 |
| 费率 | 管理费 + 托管费 + 销售服务费 | 费率高低 |
| 股票因子 | PB/ROE/营收增速/利润增速/股息率/估值分位 | 深度价值、质量成长、红利稳健 |
| 基准日收益 | 114 只有可靠源 | Alpha、Beta、超额收益、信息比率 |
| 基金规模 | 有 | 规模标签 |

硬限制：
- 持仓是单期快照，不是多期时序 → 风格稳定性只能靠多期跑批积累
- 3 年数据只有 3 只 → 长周期标签样本极少
- 28 只缺基准源 → 相对基准分析永远只能覆盖 114 只

### 4.2 六个模块

#### 模块 1：基金诊断中心——每只基金的完整体检报告

最终形态：输入一个基金代码，输出一份结构化诊断报告，包含基金画像、展示资格、正式结论、观察信号、待校准信号、风险提示、同类比较、基准解析、证据链。

当前完成度：80%。缺同类排名分位数。

#### 模块 2：FOF 子基金筛选和组合构建工作台

最终形态：一个可交互的核心-卫星组合构建器，含目标桶权重范围、单只基金上限、风险标签收紧、相关性矩阵、组合风险指标、归因。

当前完成度：50%。有角色、约束、草案、人工校准。缺相关性、组合风险、归因。

#### 模块 3：基金经理监控面板

最终形态：按基金经理维度聚合，看一个经理管了哪几只基金、业绩如何、风格是否一致。

当前完成度：20%。有经理任期标签，但没有经理维度聚合。

#### 模块 4：标签时序追踪和风险预警

最终形态：每月跑批一次，保留历史快照，展示标签随时间变化。

当前完成度：30%。有 run diff 能力（对比两次 run），但没有时间线展示和预警。

#### 模块 5：竞品横评工作台

最终形态：选一个风格或类型，横向对比同类基金，含排序、雷达图、并排对比。

当前完成度：40%。有标签筛选和搜索，但缺雷达图、排序、对比。

#### 模块 6：标准化报告输出

最终形态：一键导出结构化报告，服务不同场景。

当前完成度：40%。有 Markdown 报告，缺 Excel 和 PDF。

### 4.3 实施优先级

| 步骤 | 做什么 | 为什么先做 |
|---|---|---|
| 第一步 | 同类排名分位数 + 竞品横评页面 | 142 只内部横评是最高频需求，且数据已有 |
| 第二步 | 相关性矩阵 + 组合风险指标 | 组合草案已有，只差这一步就能给 FOF 用 |
| 第三步 | 标签时序 + 预警 | 每月跑批已能做，只缺展示和预警规则 |
| 第四步 | 经理监控面板 | 数据已有，只缺聚合视图 |
| 第五步 | 行业/风格暴露约束 | 组合层增强 |
| 第六步 | Brinson 归因 | 需要补基准成分股，工作量最大 |

## 五、当前项目卡点分析

### 卡点 1：28 只基金基准数据源缺失（数据问题，不是技术问题）

这 28 只基金的"比较尺子"需要付费数据源。比如"中债总指数"的日收益数据，免费渠道拿不到，需要向中债登购买授权。这不是写代码能解决的，是预算和采购问题。

影响：这 28 只基金暂时不能算"有没有跑赢基准"，但基础标签（收益、风险、持仓）都能算。

解决方案：
- 短期：接受 114 只可展示，28 只标记为"暂不可展示"
- 中期：申请数据采购预算，买中债总/中国债券总/标普中国债券的授权
- 备选：看是否有公司已有的数据源可以接入

### 卡点 2：缺少"同类排名"维度（功能缺失，数据已有）

现在系统只会说"这只基金年化收益 25%，达到 15% 的门槛了，贴个'收益优秀'标签"。但业务人员真正想问的是"在 114 只里排第几？是前 10% 还是前 50%？"

这个功能技术上不难（数据都有了，只是加一个分位数计算），但还没做。

影响：业务人员拿到结果后，还要手动去 Excel 里排序，体验不好。

解决方案：在跑批时增加分位数计算，前端展示排名。预计 2-3 天可完成。

### 卡点 3：组合层缺"相关性"和"组合风险"（功能缺失，数据已有）

现在组合草案会告诉你"建议核心池放 9 只基金，各放多少比例"。但它不会告诉你"这 9 只基金之间像不像？如果它们买的是同一批股票，那分散风险的效果就打折扣了"。

这个功能数据已经有了（每只基金的日收益都有），只是还没算。

影响：FOF 基金经理不敢直接用这个组合草案，因为不知道基金之间是否重叠。

解决方案：用日收益算两两相关系数，前端展示相关性矩阵。预计 3-5 天可完成。

### 卡点 4：持仓数据只有一期，风格判断"站不稳"（数据时序问题）

现在系统判断一只基金是不是"质量成长"风格，只看了最新一期持仓报告。但基金一个季度可能调一次仓，只看一期就像"只看一张照片就判断一个人的性格"。

影响：高级风格标签（深度价值、质量成长、红利稳健）只能作为"研究线索"，不能作为强结论。

解决方案：
- 短期：继续标记为"待校准"，不升级为正式结论
- 中期：每月跑批，积累多期持仓，等有 4-6 期数据后再判断风格稳定性
- 这个不是卡住，是需要时间积累

### 卡点 5：前端"最后一公里"体验不够好（产品问题）

后端能力其实挺强了，但前端页面还是"给工程师看的"。业务人员打开后看到一堆英文状态码和技术术语，不知道怎么用。

影响：系统做好了但没人用，因为"看不懂"。

解决方案：前端改造，重点是首页直接显示"142 只里 114 只可用、28 只不可用、为什么"，单基金报告先显示结论再展开证据，竞品横评页面选一个风格就能横向对比同类基金。

## 六、基准处理详解

### 6.1 什么是基准

每只基金在成立时，基金合同里都会写一句话，类似这样：

> 本基金的业绩比较基准为：沪深300指数收益率 × 80% + 中证全债指数收益率 × 20%

这句话的意思是：这只基金应该和一把"组合尺子"比。这把尺子由两部分组成：
- 80% 权重的沪深300指数（代表股票大盘）
- 20% 权重的中证全债指数（代表债券市场）

如果这只基金一年涨了 25%，但这把尺子同期涨了 30%，那它其实没有跑赢自己应该比较的对象。

### 6.2 系统具体怎么做的

**第一步：从数据库里读出基金合同写的基准原文**

系统从 `fund_profiles` 表里读取每只基金的 `benchmark` 字段。这个字段就是基金合同里写的那句话原文。比如：

```
沪深300指数收益率*80%+中证综合债指数收益率*20%
```

**第二步：把这句话拆成"组件"**

系统用代码把这句话按 `+` 号拆开，每一部分就是一个"组件"。比如上面这句拆成：

| 组件 | 权重 |
|---|---|
| 沪深300指数收益率 | 80% |
| 中证综合债指数收益率 | 20% |

**第三步：给每个组件找到对应的"指数代码"**

系统里有一张"对照表"（代码里叫 `INDEX_MAP`），大概有 60 多个映射，比如：

| 基准原文里出现的名字 | 对应的指数代码 | 从哪取日收益 |
|---|---|---|
| 沪深300指数 | 000300 | 东方财富行情接口 |
| 中证全债指数 | H11001 | 新浪财经接口 |
| 中证综合债指数 | H11009 | 本地已入库 |
| 上证国债指数 | 000012 | 东方财富行情接口 |
| 中证军工指数 | 399967 | 东方财富行情接口 |

系统拿着拆出来的组件名去这张表里找，找到就配上代码和取数来源。

**第四步：拿每天的指数收益，按权重合成一条基准日收益**

比如沪深300涨了 1%，中证综合债涨了 0.01%，权重是 80% + 20%，那当天基准收益就是：

```
1% × 80% + 0.01% × 20% = 0.802%
```

这样每天都合成一条，最后得到一条完整的基准日收益序列。基金每天的收益就可以和这条基准逐天对齐比较。

### 6.3 找不到怎么办

**情况 A：组件名能认出来，但没有日收益数据源**

比如"中债总指数"，系统认得它（代码 `LOCAL_CBOND_TOTAL`），但这个指数的日收益数据需要向中债登买授权，免费渠道拿不到。

系统的做法是：不拿别的指数代替，直接标记为"缺收益源"。这只基金就不能展示 Alpha、Beta、超额收益。这就是 28 只暂不可展示的主要原因（27 只属于这种情况）。

**情况 B：组件名是存款利率**

比如基准里写了"银行活期存款利率(税后)"，这不是市场指数，没有行情。系统会用一个固定年化利率（当前 0.35%）换算成每日收益来参与合成。

**情况 C：组件名是固定年化收益率**

比如某基金基准写了"沪深300指数收益率*95% + 2.5%(指年收益率)"。系统会把 2.5% 按交易日折算成每日收益来参与合成。这类能处理。

**情况 D：组件名完全认不出来**

比如"新华富时中国A200指数"，系统虽然知道这个名字，但 FTSE（富时）的指数数据需要授权，拿不到。标记为"缺收益源"。

### 6.4 这个做法合不合理

**合理的部分：**

1. 逐只解析，不是统一套沪深300 — 这是正确的做法。很多系统的通病就是所有基金都拿沪深300比，这会对债券占比高的基金造成严重误判。
2. 拆组件合成 — 复合基准拆成组件再按权重合成日收益，这是行业标准做法。
3. 不代理 — "中债总"拿不到就用"中债综合"代替？系统坚决不做。因为这两个指数的成分和走势不同，替代会产生错误的 Alpha 和 Beta。
4. 留审计记录 — 每个组件的解析结果、状态、原因都写进了 `benchmark_components` 表，可以复查。

**值得讨论的部分：**

1. 对照表是手工维护的 — `INDEX_MAP` 是代码里写死的 60 多个映射，不是自动从指数库查的。如果基金合同里出现了新指数名，需要人工加映射。这在 142 只规模下可行，但如果扩到几千只，维护成本会上升。
2. 部分债券指数用新浪接口取数 — 新浪财经接口不是官方数据源，稳定性和权威性不如中债登官方授权。目前能用，但生产环境建议替换为授权源。
3. 114/142 的覆盖率 — 80% 的可展示率在当前条件下算不错了，剩余 28 只确实需要授权数据源。

**总结：基准处理的逻辑和方法是对的，主要瓶颈在数据源采购，不在技术方案。**

## 七、标签体系详解

### 7.1 标签是什么

标签就是给基金贴的"特征小纸条"。比如：

- "长期收益优秀" — 这只基金过去一年涨了不少
- "夏普较高" — 承担的风险换来了不错的收益
- "行业集中度高" — 这只基金的钱集中在少数几个行业
- "经理任期较长" — 当前基金经理管了比较久

每个标签都有三个要素：
- 标签名：比如"长期收益优秀"
- 指标值：比如年化收益 25.3%
- 阈值：比如 15%（达到或超过就触发）

### 7.2 系统怎么打标签的

**第一步：算指标**

系统先把原始数据算成具体指标。比如：

| 指标 | 怎么算 | 用什么数据 |
|---|---|---|
| 年化收益 | 每天收益连乘后换算成年 | 基金每日净值增长率 |
| 年化波动 | 每天收益的标准差 × √252 | 基金每日净值增长率 |
| 最大回撤 | 从历史最高点往下跌的最大幅度 | 基金每日净值增长率 |
| 夏普 | 年化收益 ÷ 年化波动 | 上面两个算出来 |
| 前十大持仓占比 | 最新一期前 10 只股票权重相加 | 基金持仓表 |
| 第一大行业占比 | 最新一期最大行业权重 | 基金行业配置表 |
| 经理任期 | 当前经理任职天数 ÷ 365.25 | 基金经理表 |
| 综合费率 | 管理费 + 托管费 + 销售服务费 | 基金费率表 |
| Alpha | 回归模型算出来的截距项 | 基金日收益 + 基准日收益 |
| Beta | 回归模型算出来的斜率项 | 基金日收益 + 基准日收益 |
| 超额收益 | 基金日收益 - 基准日收益，连乘后年化 | 基金日收益 + 基准日收益 |
| 信息比率 | 年化超额收益 ÷ 跟踪误差 | 上面两个算出来 |

**第二步：拿指标和阈值比**

每个标签都有一个规则，规则就是"指标值达到某个门槛就触发"。所有阈值写在 `config/rules.v1.json` 里。

**第三步：记录三种状态**

| 状态 | 含义 | 举例 |
|---|---|---|
| `triggered`（触发） | 数据有，指标也达到门槛了 | 年化收益 25%，达到 15% 门槛，触发"长期收益优秀" |
| `not_triggered`（未触发） | 数据有，但指标没到门槛 | 经理任期 2 年，没到 5 年门槛，没触发"经理任期较长" |
| `not_computed`（无法计算） | 数据不够，根本没法算 | 没有费率数据，就不知道费率高低，标记为"无法计算" |

**第四步：留证据**

每个标签触发时，系统同时保存一条"证据"，包含指标名、指标值、阈值、数据来源、白话解释。

### 7.4 标签合理性评估

**合理的部分：**

1. 规则透明可查 — 每个标签的阈值写在配置文件里，不是黑箱。改阈值只需要改 JSON，不用改代码。
2. 三种状态区分 — "触发/未触发/无法计算"三分法比简单的"有/没有"更严谨。
3. 留证据链 — 每个标签都能查到指标值、阈值、来源、白话解释，可审计可复查。
4. 风格穿透 — 不是看基金名字猜风格，而是穿透到持仓股票的因子去判断。
5. 覆盖率先决条件 — 因子覆盖不够就不出正式风格标签。

**值得讨论的部分：**

1. 阈值是拍脑袋定的还是有校准依据？ — 比如年化收益 ≥ 15% 算"优秀"，这个 15% 怎么来的？目前看是人工设定的固定值，没有按市场环境动态调整。
2. 持仓只有一期 — 风格判断只看了最新一期持仓。基金可能一个季度调一次仓。
3. Alpha 用 1 年窗口 — 1 年的窗口可能受特殊市场环境影响较大。
4. 夏普无风险利率按 0 — 实际应该用一年期国债收益率（约 1.5-2%）。
5. 费率标签按固定阈值 — 主动基金和被动指数基金的合理费率范围不同，应该分层校准。

## 八、风格标签增强计划

### 8.1 当前问题

#### 问题 1：风格体系只有三个维度，太粗

现在只有"深度价值 / 质量成长 / 红利稳健"三个风格标签。但真实的权益基金风格远不止这些。95 只基金（占 67%）落在 `style_pending_rule_definition`，不是这些基金没风格，而是风格标签体系覆盖面太窄。

#### 问题 2：维度太单一，只看基本面因子

现在判定风格的因子只有 PB、ROE、营收增速、股息率、估值分位。但投研常用的风格维度至少还有动量、规模、波动率、流动性。

#### 问题 3：阈值一刀切，不区分基金类型

深度价值 60% 的门槛，对主动基金和指数基金应该不同。一只银行 ETF 自然 90% 以上都是深度价值，但一只主动管理基金如果有 40% 深度价值股票已经很有特色了。

### 8.2 已有因子使用情况

| 已有因子 | 现在用了吗 | 应该用来做什么 |
|---|---|---|
| `pb`（市净率） | 用了，但只配合估值分位判深度价值 | 可以单独做"低估值"标签 |
| `pe`（市盈率） | **读了但完全没用** | 应该加入估值判定 |
| `roe`（净资产收益率） | 用了，配合营收增速判质量成长 | 可以单独做"高盈利"标签 |
| `revenue_growth`（营收增速） | 用了 | — |
| `profit_growth`（利润增速） | **读了但完全没用** | 应该用来做"利润高增长"标签 |
| `dividend_yield`（股息率） | 用了 | — |
| `valuation_percentile`（估值分位） | 用了 | — |
| `market_cap_bucket`（市值分档） | **读了但完全没用** | 应该用来做"大盘/中盘/小盘"标签 |
| `log10_market_cap`（对数市值） | **读了但完全没用** | 同上 |
| `close_price`（收盘价） | **读了但完全没用** | 不直接用于标签，可辅助计算 |

**最大浪费：系统已经读取了 PE、市值、利润增速这些因子，但标签规则完全没用它们。**

| `mixed_style`（混合风格） | 3 类以上风格权重都在 15%-35% | 什么都买一点，没有明确风格 |

### 8.4 预期效果

做完前 3 步，142 只基金里 `style_pending_rule_definition`（风格未定）的 95 只，预计能降到 30-40 只。

做完 5 步，风格标签体系就从"3 个粗标签"升级成"20+ 个精细标签 + 3 层下钻"，基本覆盖专业投研常用的风格维度。

### 8.5 优先级总结

| 步骤 | 做什么 | 需要新数据吗 | 工作量 |
|---|---|---|---|
| 第 1 步 | 用已有因子补 7 个标签 | 不需要 | 小 |
| 第 2 步 | 用已有行业映射补 6 个行业主题标签 | 不需要（需扩分组） | 中 |
| 第 3 步 | 前端做下钻展示 | 不需要（数据已有） | 中 |
| 第 4 步 | 补动量和波动率因子 + 4 个标签 | 需要加因子抓取 | 大 |
| 第 5 步 | 做风格组合标签 | 不需要 | 小 |

## 九、"下钻"概念详解

### 9.1 什么是下钻

"下钻"（Drill-down），大白话就是从大结论往小细节一层层点进去看。

打比方：打开外卖 App 看一个餐厅评分 4.5 分，这是"大结论"。点进去看口味 4.8、服务 4.2、速度 3.5，这是"第一层下钻"。再点"口味 4.8"看到 120 条评价的分布，这是"第二层下钻"。

### 9.2 放到我们项目里

现在系统告诉你："基金 000020 贴了 quality_growth 质量成长标签"。这是大结论。

如果系统能"下钻"：

```text
第一层：基金 000020 → 质量成长标签
  原因：质量成长股票持仓权重 45%，达到 50% 阈值

  ↓ 下钻到第二层

第二层：这 45% 是由哪些股票贡献的？
  贵州茅台  权重 9.5%  ROE 30%  营收增速 18%  ✓ 命中
  五粮液    权重 7.2%  ROE 25%  营收增速 15%  ✓ 命中
  泸州老窖  权重 5.8%  ROE 22%  营收增速 21%  ✓ 命中
  招商银行  权重 8.0%  ROE 16%  营收增速 5%   ✗ 营收增速不达标

  ↓ 下钻到第三层

第三层：招商银行为什么没命中？
  ROE = 16%（≥ 15% ✓ 达标）
  营收增速 = 5%（< 15% ✗ 不达标）
  结论：营收增速不够，不算质量成长股票
```

### 9.3 当前下钻能力

代码里已经有 `fund_equity_style_contributions` 表，记录了每只股票对每个风格标签的贡献明细，包括：
- 股票代码
- 持仓权重
- 命中了哪个风格
- 命中时的因子值是多少
- 用的规则阈值是多少

但目前这个下钻数据只落库了，没有在前端展示出来。业务人员在网页上只看到"质量成长"这个标签，看不到下面那层股票明细。这就是要补的。

>

### 10.2 风格标签汇报

> 现在风格标签只有深度价值、质量成长、红利稳健三个，太粗了，67% 的基金打不上标签。
>
> 我分五步走：第一步把已有但没用起来的因子用上，补低估值、大盘小盘、利润高增长这些标签，不需要新数据；第二步用已有的行业映射数据补科技、金融、消费等行业主题标签；第三步把下钻展示做出来——现在每只股票对风格的贡献明细已经落库了，但前端没展示，点一个风格标签就能看到是哪些股票贡献的、每只股票的因子值是多少、为什么命中或没命中；第四步补动量和波动率两个新因子；第五步做风格组合标签，比如"价值红利""大盘成长"。
>
> 前三步不需要新数据源，做完后风格未定的基金预计从 95 只降到 30-40 只。

### 10.3 基准处理汇报

> 我们不是拿所有基金和沪深300比。每只基金在基金合同里都写了"我应该和谁比"，比如"沪深300 × 80% + 中证全债 × 20%"。我们做的就是把这句话拆开，找到每个组件对应的指数代码，拿每天的指数收益按权重合成一条基准日收益。这样做比统一套沪深300准确得多。目前 142 只里有 114 只找到了完整的基准日收益数据，28 只因为某些指数数据需要付费授权暂时没法算。我们宁可不算也不用错误的指数代替，因为那样算出来的 Alpha 和 Beta 会误导判断。

### 10.4 标签体系汇报

> 我们给每只基金打了几十个标签，比如"长期收益优秀""夏普较高""行业集中度高""经理任期较长"。每个标签都是一个规则判断：先算出某个指标（比如年化收益 25.3%），再和一个阈值比（比如 15%），达到了就触发标签。同时我们会记录三种状态：触发、未触发、无法计算。每个标签都留了证据——指标是多少、阈值是多少、数据从哪来。风格类标签更复杂，会穿透到基金买的每只股票去看它的 PB、ROE、股息率等因子，再按持仓权重汇总判断这只基金整体是什么风格。
