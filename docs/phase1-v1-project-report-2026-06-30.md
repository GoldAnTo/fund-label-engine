## 1. 先用一句人话说明这个项目

这个项目不是“推荐买哪只基金”的系统，也不是自动交易系统。

它做的事情更像一个“基金体检和标签报告系统”：

```text
拿到一批基金
  -> 先检查数据够不够
  -> 再计算收益、风险、持仓、经理、费率、风格等指标
  -> 再按固定规则给基金打标签
  -> 每个标签都保存证据
  -> 最后告诉用户：哪些基金可以展示，哪些暂时不能展示，原因是什么
```

换句话说，它不是替人做最终投资决定，而是把基金研究里最容易混乱的部分变得可解释、可复查、可重复。

当前 Phase1 v1 的结论可以先记住三组数字：

| 数字 | 含义 |
|---|---|
| 142 | 第一版正式纳入分析的基金数量 |
| 114 | 其中可以正式展示相对基准表现的基金数量 |
| 28 | 暂时不能展示相对基准表现的基金数量 |

这里的“相对基准表现”，可以先理解为：这只基金有没有跑赢它自己应该比较的那把尺子。

## 2. 为什么需要这个项目

如果只看一只基金的涨跌，很容易误判。

例如一只基金一年涨了 20%，听起来不错。但如果它的业绩比较基准同期涨了 30%，那它其实没有跑赢自己的赛道。反过来，一只基金只涨了 5%，如果它对应的基准跌了 10%，那它可能反而表现不错。

所以基金研究不能只问：

```text
这只基金涨了吗？
```

更应该问：

```text
它的数据够不够？
它应该和谁比？
它跑赢了自己的基准吗？
它的风险是不是也更高？
它为什么被打上某个标签？
如果没打标签，是规则不满足，还是数据不够？
```

`fund-label-engine` 的目标就是把这些问题拆成清楚的机器流程，让每个结论都有证据。

## 3. 项目最终产出是什么

对每只基金，系统会产出几类东西：

| 产出 | 给非专业读者的解释 |
|---|---|
| 数据覆盖率 | 这只基金的数据够不够算 |
| 特征值 | 具体指标，比如收益、回撤、费率、持仓集中度 |
| 标签 | 对基金特征的总结，比如“数据充足”“权益仓位高”“Alpha 为正” |
| 证据 | 为什么打这个标签，指标是多少，阈值是多少 |
| 计算状态 | 这个标签是触发、未触发，还是因为缺数据没法算 |
| 分类 | 这只基金属于权益类、主动管理还是被动指数 |
| 分组 | 它进入哪个业务池，比如可计算池、主动权益候选池、被动工具池 |
| 可展示判断 | 哪些结果可以正式展示，哪些只能观察，哪些暂时不能展示 |

最重要的是：系统不是只给一个结论，而是保留“为什么”的过程。

## 4. 为什么第一版只选这些基金

### 4.1 基金类型不一样，不能用同一套规则

基金不是一种东西。股票基金、债券基金、货币基金、QDII、FOF 的研究方式都不同。

举几个例子：

- 股票型基金：重点看股票持仓、行业、风格、收益风险。
- 债券基金：重点看久期、信用等级、债券品种、杠杆。
- 货币基金：重点看流动性、七日年化、规模稳定性。
- FOF：重点看它买了哪些底层基金。
- QDII：还要考虑海外市场和汇率。

如果第一版强行全都做，就会出现“拿股票基金尺子去量债券基金”的问题。

所以第一版只做权益相关基金，也就是更适合用持仓、行业、股票因子、相对股票/股债基准来解释的一批基金。

第一版纳入的类型：

| 类型 | 为什么纳入 |
|---|---|
| 股票型 | 主要投资股票，适合做权益标签 |
| 混合型-偏股 | 股票占比较高，也适合权益分析 |
| 混合型-灵活 | 可能有较高股票仓位，需要通过数据 gate 再确认 |
| 指数型-股票 | 可以作为被动指数工具单独识别 |

第一版暂不纳入：

- 纯债基金。
- 货币基金。
- QDII。
- FOF。
- 商品基金。
- REITs。

这些不是不重要，而是要另做标签体系。

### 4.2 系统到底怎么确认“这只基金在不在第一版范围”

这里不要理解成“人工凭感觉挑了 142 只”。系统实际用了三层判断。

第一层：看结构化基金类型。

数据源里每只基金都有 `fund_profiles.fund_type`，也就是基金类型字段。系统只接受下面四个值：

| 接受的类型 | 为什么接受 |
|---|---|
| `股票型` | 主要解释对象就是股票持仓、收益风险、行业和风格 |
| `混合型-偏股` | 股票占比通常较高，适合进入权益标签体系 |
| `混合型-灵活` | 可能有较高股票仓位，但后面还要用持仓数据再确认 |
| `指数型-股票` | 可以作为被动指数工具识别和比较 |

如果 `fund_type` 是债券型、货币型、QDII、FOF、商品、REITs 等，第一版不会硬算。原因很简单：这些基金应该用另一套尺子，例如债券基金要看久期、信用等级，货币基金要看流动性，QDII 要看海外市场和汇率。

第二层：看 Phase1 正式清单。

项目还有一个业务确认过的清单：

```text
data/phase1_fund_codes_v1_official.txt
```

跑批时会读取这个清单，只保留清单里的基金代码。也就是说，基金必须同时满足：

```text
基金类型属于四类之一
并且
基金代码在 Phase1 v1 正式清单里
```

才会进入本次正式报告口径。

第三层：进入计算前再做数据 gate。

前两层只是确认“这只基金理论上适合第一版”。进入计算前，还要检查它的数据够不够。如果类型对，但数据缺失，也不能输出正式结论。

所以完整逻辑是：

```text
先按基金类型圈大范围
再用 Phase1 正式清单锁定业务范围
最后用数据 gate 判断能不能正式计算
```

这就是为什么早期 168 只候选基金后来收敛成 142 只正式基金。不是因为另外 26 只一定不好，而是因为第一版要保证“范围清楚、数据够用、结果能解释”。

当前 142 只基金的类型分布：

| 基金类型 | 数量 |
|---|---:|
| 混合型-灵活 | 71 |
| 混合型-偏股 | 41 |
| 股票型 | 16 |
| 指数型-股票 | 14 |
| 合计 | 142 |

## 5. 这 142 只基金具体跑了什么

可以把一次跑批理解成一次流水线体检。代码里有些计算会在同一个步骤里完成，但为了让非专业读者看懂，下面按产品逻辑拆成 8 步。

```text
第 1 步：加载原始材料
第 2 步：确认基金范围
第 3 步：检查数据够不够
第 4 步：计算基础指标
第 5 步：计算持仓风格
第 6 步：计算相对基准指标
第 7 步：按规则打标签
第 8 步：生成分类、分组和展示状态
```

下面逐步解释。

### 5.1 第 1 步：加载这只基金的原始材料

系统不是凭基金名字判断，也不是让大模型自由猜，而是先从数据库里把这只基金的“材料袋”拿出来。

每只基金会加载这些材料：

| 材料 | 来自哪里 | 用来做什么 |
|---|---|---|
| 基金代码、名称、类型、规模 | `fund_profiles` | 确认身份、范围、规模标签 |
| 每日收益率 | `nav_history.daily_growth_rate` | 计算收益、波动、回撤、夏普 |
| 最新一期股票持仓 | `stock_holdings` | 计算持仓集中、权益仓位、持仓风格 |
| 最新一期行业配置 | `industry_allocations` | 计算行业集中或分散 |
| 基金经理任期 | `fund_manager_links.tenure_days` | 判断经理任期是否较长 |
| 管理费、托管费、销售服务费 | `fee_structures` | 判断费率高低 |
| 股票因子 | `stock_factor_values` 或 `stock_factors` | 计算深度价值、质量成长、红利稳健等风格 |
| 基准日收益 | `benchmark_returns` | 计算超额收益、Alpha、Beta、信息比率 |

这里有两个重要细节。

第一，股票持仓和行业配置都取“最新一期报告期”。比如一只基金有多个季度持仓，系统先找到最新 `report_period`，再取那一期的股票明细。

第二，ETF 场内占位费率不会被当作真实费率。有些 ETF 数据里只有“场内ETF-无费率信息”这种占位记录，系统不会把它理解成费率为 0，而是继续视为费率证据不足。这样做是为了避免把“没抓到数据”误判成“费率很低”。

### 5.2 第 2 步：确认基金范围

系统先看基金类型是不是下面四类之一：

- 股票型。
- 混合型-偏股。
- 混合型-灵活。
- 指数型-股票。

如果不是，就不进入第一版正式标签范围。

如果是，还要看基金代码是不是在 Phase1 v1 正式清单里。这样可以避免全库 1 万多只基金都被拉进来陪跑，导致报告口径混乱。

确认范围后，系统还会做一个分类：

| 分类 | 判断方式 | 人话解释 |
|---|---|---|
| `equity_related` | 基金类型属于四类之一 | 第一版权益相关基金 |
| `passive_index` | 类型或名称里有指数、ETF、联接、INDEX | 更像被动指数工具 |
| `active` | 未命中指数/ETF/联接特征 | 按主动管理候选处理 |

注意，`passive_index` 不是坏标签，只是说明它应该和主动基金分开看。

### 5.3 第 3 步：检查数据够不够

这一步叫数据 gate。可以理解成进考场前先检查准考证和证件。

数据 gate 检查的是“这只基金有没有资格进入正式计算”。它不是判断基金好坏，而是判断证据够不够。

当前 Phase1 v1 跑批使用的关键门槛来自本次运行的规则快照：

| 检查项 | 本次门槛 | 为什么要检查 | 不通过会怎样 |
|---|---:|---|---|
| 基金类型 | 必须是四类之一 | 防止拿错标签体系 | 不进入第一版正式范围 |
| NAV 日收益样本 | 至少 180 条 | 至少能支撑约 1 年收益风险判断 | 收益风险标签不能正式算 |
| 股票持仓条数 | 至少 1 条 | 没有持仓就不能看集中度和风格 | 持仓类标签不能正式算 |
| 股票持仓总权重 | 至少 50% | 避免只穿透到一点点股票就乱下结论 | 风格和持仓判断降级 |
| 行业配置 | 至少 1 条 | 没有行业数据就不能判断行业集中 | 行业标签不能正式算 |
| 基金经理任期 | 不能是空 | 没有任期就不能判断经理稳定性 | 经理标签不能正式算 |
| 费率结构 | 管理费和托管费都要有 | 费率低/高必须有真实费用证据 | 费率标签不能正式算 |
| 基金规模 | 不能是空 | 规模过小和适中都要有规模数字 | 规模标签不能正式算 |
| 权益仓位 | 不能是空 | 要知道股票仓位水平 | 权益仓位标签不能正式算 |

当前 142 只基金在上述基础 gate 上全部通过，所以都进入 `label_ready_pool`。

但这里还有一个非常重要的区别：

```text
基础数据够用
不等于
所有高级指标都能展示
```

比如一只基金基础数据都齐，但它的基准缺少可靠日收益源，那它仍然不能展示 Alpha、Beta、超额收益。这个区别就是为什么当前有 142 只 `label_ready`，但只有 114 只 `relative_label_ready`。

如果 gate 不通过，系统不会直接丢掉这只基金，而是会做三件事：

1. 给它打 `data_insufficient`，意思是数据不足。
2. 给它打 `manual_review_required`，意思是需要人工复核。
3. 把依赖缺失数据的标签降级成观察状态，不当作正式结论。

系统还会把每个失败原因写下来。例如：

```text
nav_returns: nav_samples_below_min
stock_holdings: stock_holdings_total_weight_low
fee_structure: fee_structure_missing
```

这对产品很重要，因为业务人员看到的不是一句“不能算”，而是知道“为什么不能算、缺什么、下一步补哪里”。

### 5.4 第 4 步：计算基础指标

基础指标可以理解成“体检里的血压、心率、身高体重”。后面的标签基本都从这些指标来。

收益风险指标来自每日收益率，系统会按几个窗口计算：

| 窗口 | 取多少个交易日 | 最少需要多少个有效样本 | 作用 |
|---|---:|---:|---|
| 1m | 最近 21 天 | 15 天 | 观察短期情况 |
| 3m | 最近 63 天 | 40 天 | 观察季度情况 |
| 1y | 最近 252 天 | 180 天 | 正式收益风险标签可用 |
| 3y | 最近 756 天 | 500 天 | 优先用于更长周期判断 |

正式标签优先用 3 年窗口；如果 3 年不够，再用 1 年窗口。当前 142 只基金都有 1 年窗口，其中 3 只也有 3 年窗口。

几个核心指标这样算：

| 指标 | 系统怎么算 | 小白怎么理解 |
|---|---|---|
| 区间收益 | 每天收益连续相乘后减 1 | 这段时间总共涨跌多少 |
| 年化收益 | 把区间收益换算成一年口径 | 方便不同长度数据之间比较 |
| 年化波动 | 日收益的波动程度乘以 `sqrt(252)` | 净值上下晃得有多厉害 |
| 最大回撤 | 从历史高点往下跌的最大幅度 | 中间最难受的时候跌了多少 |
| 夏普 | 年化收益 / 年化波动 | 每承担一份波动，大概换来多少收益 |
| 前十大持仓占比 | 最新一期前 10 只股票权重相加 | 是不是押得比较集中 |
| 第一大行业占比 | 最新一期最大行业权重 | 是不是集中在一个行业 |
| 权益仓位 | 最新一期股票持仓权重相加 | 股票资产占基金净值多少 |
| 经理任期 | 最大任职天数 / 365.25 | 当前经理大约管了几年 |
| 综合费率 | 管理费 + 托管费 + 销售服务费 | 持有成本大概高不高 |
| 基金规模 | `fund_profiles.asset_size` | 基金大概有多大 |

这些指标再进入标签规则。

例如：

| 标签 | 触发规则 | 小白理解 |
|---|---|---|
| `long_term_return_strong` | 年化收益 >= 15% | 收益表现达到阈值 |
| `volatility_high` | 年化波动 >= 30% | 波动比较大 |
| `volatility_low` | 年化波动 <= 12% | 波动比较小 |
| `drawdown_high` | 最大回撤 <= -20% | 曾经跌得比较深 |
| `sharpe_high` | 夏普 >= 1.0 | 风险收益比达到阈值 |
| `holding_concentration_high` | 前十大持仓 >= 55% | 持仓比较集中 |
| `industry_concentration_high` | 第一大行业 >= 60% | 行业非常集中 |
| `industry_concentration_observe` | 第一大行业 45% 到 60% | 行业集中，需要观察 |
| `industry_diversified` | 第一大行业 < 20% 且行业数 >= 5 | 行业比较分散 |
| `equity_position_high` | 权益仓位 >= 80% | 股票仓位较高 |
| `manager_tenure_long` | 经理任期 >= 5 年 | 经理任期较长 |
| `fee_low` | 综合费率 <= 1.2% | 费率较低 |
| `fee_high` | 综合费率 > 2.5% | 费率偏高 |
| `fund_size_small` | 规模 < 1 亿元 | 规模偏小 |
| `fund_size_moderate` | 5 亿到 100 亿元 | 规模处于适中区间 |

这里也要注意：没有触发某个标签，不等于这只基金差。它只说明“这个指标没有达到这条规则的阈值”。

### 5.5 第 5 步：计算持仓风格

持仓风格不是看基金名字，也不是看基金经理宣传语，而是看它实际买了哪些股票。

系统做持仓风格时分四小步。

第一小步：把基金持仓和股票因子对上。

基金持仓里有：

```text
股票代码、股票名称、持仓权重
```

股票因子里有：

```text
PB、ROE、营收增速、利润增速、股息率、估值分位数
```

系统用股票代码把两边匹配起来。匹配上以后，就知道“基金持有的这只股票是不是低估值、是不是高 ROE、是不是高股息”。

第二小步：先检查因子覆盖率。

不是所有持仓股票都一定有完整股票因子，所以系统会算：

```text
因子覆盖权重 = 有股票因子的持仓权重合计
```

本次规则是：

| 因子覆盖权重 | 系统处理 |
|---:|---|
| 低于 50% | 不输出正式风格标签 |
| 50% 到 70% | 只输出观察，不输出正式风格标签 |
| 高于或等于 70% | 可以进入正式风格判断 |

当前 142 只基金里，136 只因子覆盖权重达到 70% 以上，6 只处在 50% 到 70% 的观察区间。

第三小步：按股票特征给每只股票归类。

系统目前主要看三类底层股票特征：

| 股票特征 | 单只股票怎么命中 | 人话解释 |
|---|---|---|
| 深度价值股票 | `PB <= 1.5` 且估值分位数 `<= 30%` | 估值比较低 |
| 质量成长股票 | `ROE >= 15%` 且营收增速 `>= 15%` | 盈利质量和增长都不错 |
| 红利稳健股票 | 股息率 `>= 3%` | 分红收益率较高 |

然后系统按基金持仓权重汇总。

例如一只基金持有 10 只股票，其中 4 只符合质量成长条件，这 4 只股票权重加起来是 45%，那么：

```text
quality_growth_weight = 45%
```

本次运行使用的风格触发门槛是：

| 风格标签 | 基金级触发门槛 |
|---|---:|
| `deep_value` | 深度价值股票权重 >= 40% |
| `quality_growth` | 质量成长股票权重 >= 40% |
| `dividend_steady` | 红利稳健股票权重 >= 50% |

第四小步：对红利风格继续拆分。

红利不是一种完全相同的风格。有些红利来自银行、能源、公用事业；有些来自消费龙头。它们的业务含义不同，所以系统会继续看红利贡献股票的行业。

规则是：

| 情况 | 输出 |
|---|---|
| 红利股票行业映射覆盖不足 70% | 保留 `dividend_steady`，并追加 `sector_mapping_insufficient` 观察标签 |
| 金融 + 能源/公用事业等红利贡献 >= 60% | 输出 `high_dividend_financial` |
| 消费行业红利贡献 >= 60% | 输出 `consumer_quality` |
| 都不满足 | 保持 `dividend_steady` |

当前高级风格触发数量：

| 标签 | 数量 | 怎么理解 |
|---|---:|---|
| `quality_growth` | 9 | 持仓里质量成长股票权重达到门槛 |
| `deep_value` | 3 | 持仓里低估值股票权重达到门槛 |
| `dividend_steady` | 2 | 持仓里高股息股票权重达到门槛 |
| `high_dividend_financial` | 1 | 红利贡献主要来自金融/能源/公用事业等 |
| `consumer_quality` | 0 | 当前没有基金达到消费质量拆分门槛 |

这一块可以作为研究线索，但不要过早包装成强结论。原因有三个：

1. 股票因子本身还需要持续校验。
2. 基金持仓是按季披露，不是每天披露。
3. 风格要看多期稳定性，不能只看一期就说基金长期风格固定。

### 5.6 第 6 步：计算相对基准指标

相对基准指标就是看基金有没有跑赢“它自己应该比较的那把尺子”。这部分后面第 6 到第 9 章会详细解释。

这里先记住三个原则：

1. 不是所有基金都统一和沪深300比。
2. 基准要逐只解析，能拆组件就拆组件。
3. 基准日收益源不可靠，就不展示 Alpha、Beta、超额收益。

当前 142 只基金里，114 只可以展示相对基准指标，28 只暂时不能展示。

### 5.7 第 7 步：按规则打标签

每个标签都不是一句主观评价，而是一个规则判断。

系统对每个标签会记录三种计算状态：

| 状态 | 含义 | 举例 |
|---|---|---|
| `triggered` | 条件满足，标签触发 | 年化收益达到 15%，触发收益优秀 |
| `not_triggered` | 数据有，但没到阈值 | 经理任期 2 年，没到 5 年 |
| `not_computed` | 前置数据不够，不能算 | 没有费率，就不能算费率高低 |

这个设计是为了避免误读。

```text
没触发
不等于
不能算
也不等于
表现不好
```

比如：

- `not_triggered`：有数据，只是没达到规则。
- `not_computed`：连数据都不够，不能下结论。
- `triggered`：达到规则，但仍然不是投资建议。

### 5.8 第 8 步：生成分类、分组和展示状态

最后系统会把标签整理成更容易给业务使用的结构。

分类回答的是：

```text
这只基金是什么类型？
```

例如：

- `equity_related`：权益相关基金。
- `active`：主动管理候选。
- `passive_index`：被动指数工具。
- `label_ready`：基础标签可以计算。
- `style_pending`：风格还需要观察。
- `style_clear`：已有较明确风格。

分组回答的是：

```text
这只基金应该放进哪个业务池子里看？
```

例如：

- `label_ready_pool`：标签可计算池。
- `active_equity_candidate_pool`：主动权益候选池。
- `passive_tool_pool`：被动指数工具池。
- `industry_concentration_watch`：行业集中观察池。
- `style_factor_ready_pool`：风格因子可用池。

展示状态回答的是：

```text
这个结论能不能正式给用户看？
```

当前最重要的展示边界是：

```text
142 只基础标签可展示
114 只相对基准标签可展示
28 只相对基准标签暂不可展示
```

这样设计的价值是：普通用户看到的不是一堆难懂指标，而是先知道“能不能展示、为什么能、为什么不能、下一步缺什么”。

## 6. 什么是“基准”，为什么它很重要

基准可以理解为“这只基金应该比较的参照物”。

举个简单例子：

- 一个沪深300指数增强基金，应该主要和沪深300比。
- 一个 80% 股票 + 20% 债券的混合基金，应该和类似 80% 股票指数 + 20% 债券指数的复合基准比。
- 一个消费主题基金，不能简单拿普通沪深300完全替代消费主题指数。

如果基准选错了，后面的超额收益、Alpha、Beta 都会错。

所以这个项目把基准处理得很严格，不是所有基金都套一个沪深300。

## 7. 基准怎么选

系统按下面顺序选基准。

### 7.1 优先看跟踪标的

如果基金本身有明确跟踪标的，比如某个指数基金跟踪沪深300，就优先用这个跟踪标的。

### 7.2 再看业绩比较基准

很多主动基金没有跟踪标的，但会披露业绩比较基准。

例如：

```text
沪深300指数收益率 * 80% + 中证全债指数收益率 * 20%
```

这说明它大致是一个股债复合比较尺子。

### 7.3 把复合基准拆成组件

系统会把基准拆开：

```text
沪深300 80% + 中证全债 20%
```

拆成：

| 组件 | 权重 |
|---|---:|
| 沪深300 | 80% |
| 中证全债 | 20% |

然后每天合成一条基准日收益：

```text
基准当天收益 = 沪深300当天收益 * 80% + 中证全债当天收益 * 20%
```

这样基金每天的收益就可以和自己的基准每天对齐比较。

### 7.4 对存款利率做合成

有些基金基准里会写：

```text
同期银行活期存款利率(税后)
```

这种没有市场指数行情，系统会用固定年化利率换算成每日收益，再参与复合基准。

### 7.5 不乱用替代基准

这是项目里非常重要的一条规则：

```text
宁可标记为缺失，也不用错误基准替代。
```

例如：

- `沪深300金融地产行业指数` 不能直接当成普通 `沪深300`。
- `沪深300安中动态策略指数` 不能直接当成普通 `沪深300`。
- `中债总`、`中国债券总`、`标普中国债券` 没有可靠日收益源时，不用其他债券指数凑数。

这会让可展示数量少一些，但能保证结论可信。

## 8. 基准质量怎么判断

基准处理完后，系统给每只基金一个基准状态。

| 状态 | 人话解释 | 能不能展示相对收益 |
|---|---|---|
| `ready` | 基准组件都认得，也有日收益数据 | 能 |
| `missing_source` | 组件认得，但缺可靠日收益 | 不能 |
| `mapping_required` | 需要人工确认精确映射 | 不能 |
| `unresolved` | 系统暂时解析不了这个基准 | 不能 |
| `benchmark_missing` | 基金没有披露基准 | 不能 |

当前 142 只基金的基准质量：

| 状态 | 基金数 |
|---|---:|
| `ready` | 114 |
| `missing_source` | 27 |
| `benchmark_missing` | 1 |

所以当前可以正式展示相对基准标签的是 114 只，另外 28 只暂时不能展示。

这 28 只不是“基金不好”，而是“基准证据还不够完整，不能严肃地算相对基准表现”。

## 9. 怎么判断有没有跑赢基准

这一部分有几个专业词。先翻译成人话：

| 专业词 | 人话解释 |
|---|---|
| 超额收益 | 基金比自己的基准多赚了多少 |
| 跟踪误差 | 基金每天和基准偏离得有多大 |
| 信息比率 | 每承担一份偏离，换来了多少超额收益 |
| Beta | 基金对基准涨跌的敏感程度 |
| Alpha | 扣掉基准影响后，基金自身多出来的收益能力 |

### 9.1 先按日期对齐

系统先把基金日收益和基准日收益按日期对齐。

如果基金有 256 天收益，基准只有 241 天收益，就按共同可用的 241 天来算。

当前 1 年窗口要求至少 180 个有效样本。Phase1 v1 中没有 NAV 样本不足的问题。

### 9.2 超额收益怎么来

每天先算：

```text
基金当天收益 - 基准当天收益 = 当天主动收益
```

再把每天主动收益连乘起来，换算成年化。

规则是：

```text
年化超额收益 >= 5%
```

就触发：

```text
excess_return_strong（超额收益较强）
```

当前 142 只里，有 66 只触发这个标签。

### 9.3 信息比率怎么来

如果一只基金跑赢基准很多，但每天波动也特别大，那它的超额收益不一定稳定。

所以系统会看：

```text
信息比率 = 年化超额收益 / 跟踪误差
```

可以简单理解为：

```text
跑赢基准的效率
```

规则是：

```text
信息比率 >= 0.5
```

当前有 56 只触发 `information_ratio_high`。

### 9.4 Alpha 和 Beta 怎么看

Beta 说明基金跟基准一起涨跌的敏感度。

- Beta 高：基金跟基准涨跌关系强，波动可能更贴近或放大基准。
- Beta 低：基金对基准涨跌不那么敏感。

Alpha 说明扣掉基准影响后，基金有没有额外收益。

规则：

| 标签 | 规则 |
|---|---|
| `alpha_positive` | Alpha >= 3% |
| `beta_high` | Beta >= 1.2 |
| `beta_low` | Beta <= 0.8 |

当前结果：

| 标签 | 基金数 |
|---|---:|
| `alpha_positive` | 88 |
| `beta_low` | 105 |
| `beta_high` | 1 |

注意：`beta_low` 不等于“好基金”。它只说明这只基金对基准涨跌不敏感，仍然要结合收益、回撤、行业、风格一起看。

## 10. 用一只基金讲完整流程

以 `000020 景顺长城品质投资混合A` 为例。

### 10.1 它的基准是什么

系统解析到它的基准是：

```text

```

也就是：

| 沪深300 | 80% |
| 中证全债 | 20% |

这只基金的解释不能只说“Alpha 很高，所以很好”。更准确的说法是：

```text
它在当前 1 年窗口里相对自己的股债复合基准表现突出，
但行业集中度较高，风格标签仍需观察，

```

```text
这只基金应该放到哪个池子里看？
```

## 14. 哪些结果可以正式展示

当前可以正式展示的主要是：

- 数据是否充足。
- 收益风险标签。
- 相对基准标签，但只限 114 只 `relative_label_ready` 基金。
- 基金经理任期。
- 费率和规模。
- 分类和业务分组。
- 每个标签的证据。

这些结果的数据来源和规则比较明确，可以作为 v1 展示口径。

## 15. 哪些结果只能观察

下面这些结果有参考价值，但要放在观察区：

- 行业集中。
- 持仓集中。
- 权益仓位高。
- 高收益高回撤观察。
- 风格稳定/漂移观察。

原因是这些结果受持仓披露时点、行业分类口径、持仓穿透深度影响较大。

## 16. 哪些结果还要校准

高级风格标签还要继续校准：

- 深度价值。
- 质量成长。
- 红利稳健。
- 金融高股息。
- 消费质量。

当前它们已经能算，但样本数还少，且需要更多历史报告期验证。

所以更适合说：

```text
系统已经具备风格识别雏形，但高级风格标签暂时作为研究线索，不作为强结论。
```

## 17. 当前最大的缺口是什么

最大的缺口不是“标签不够多”，而是“部分基金基准还不够完整”。

28 只暂不可展示基金的原因：

| 原因 | 数量 | 该怎么处理 |
|---|---:|---|
| 缺少可靠基准收益源 | 27 | 补中债总、中国债券总、标普中国债券、H30124、服务业、FTSE/MSCI/定制 ESG 等收益源 |
| 未披露业绩基准 | 1 | 保持不可展示或人工补充基准 |

因此下一步最有价值的是补基准源和精确映射，而不是继续增加一堆新标签。

## 18. 这个项目的边界

### 18.1 不做投资建议

系统不会输出：

- 买入。
- 卖出。
- 申购。
- 赎回。
- 排名推荐。

它只输出标签和证据。

### 18.2 不做全市场万能标签

Phase1 v1 只覆盖 142 只权益相关基金。

债券、货币、QDII、FOF、商品、REITs 后续要另开标签体系。

### 18.3 不把所有标签当成同一等级

标签必须分层：

| 层级 | 含义 |
|---|---|
| 正式结论 | 可以进入 v1 展示 |
| 观察信号 | 有参考价值，但要提示谨慎 |
| 待校准信号 | 技术上能算，但暂不做强结论 |
| 阻断原因 | 数据或基准不够，不能展示 |

如果工作台把所有标签混在一起，非专业用户很容易误读。

## 19. 下一步建议

### 19.1 产品上

把工作台首页从“批次列表”改成“可展示池”：

- 一打开就看到 142/114/28。
- 能看到 28 只为什么不能展示。
- 能看到哪些缺口影响基金最多。
- 单基金报告先讲“能不能展示”，再讲指标明细。

### 19.2 数据上

## 21. 组合工作台：从标签到组合草案

前面 1 到 20 章讲的是"基金标签和证据系统"。从这一章开始，讲项目在标签之上新增的一层：组合工作台。

### 21.1 为什么需要组合层

标签回答的是"这只基金有什么特征"。但产品池管理还需要回答另一个问题：

```text
这只基金在组合里应该扮演什么角色？
能放多少权重？
哪些基金可以一起进核心池，哪些只能做卫星？
```

所以项目在标签引擎之上增加了一个组合层。它的定位是：

```text
标签引擎 = 证据层（算指标、打标签、存证据）
组合层   = 衍生层（基于标签推导角色、约束权重、生成草案）
```

组合层不做交易，不做买卖建议，也不做自动准入。它只做一件事：把已有的标签结果转换成可解释的组合角色和权重草案，供研究员审核。

### 21.2 角色推导

系统根据基金已有的标签和分组，自动推导每只基金在组合里的角色。

当前角色定义在 `config/portfolio_roles.v1.json`，主要角色如下：

| 角色 | 人话解释 | 推导条件（简述） |
|---|---|---|
| `core_holding_candidate` | 核心候选 | 主动权益 + 数据充足 + 经理任期长 + 无高回撤 |
| `satellite_alpha` | 卫星阿尔法 | Alpha 为正，且有信息比率或超额收益证据 |
| `defensive_anchor` | 防守锚 | Beta 低或波动低，且无高回撤 |
| `index_tool` | 指数工具 | 被动指数或指数型基金 |
| `low_cost` | 低成本 | 费率较低标签触发 |
| `active_equity_candidate` | 主动权益候选 | 在主动权益候选池里 |
| `style_quality_growth` | 质量成长风格 | 正式质量成长风格标签触发 |
| `style_deep_value` | 深度价值风格 | 正式深度价值风格标签触发 |
| `style_dividend_steady` | 红利稳健风格 | 正式红利稳健风格标签触发 |
| `needs_review` | 需复核 | 数据不足或需人工复核 |

一只基金可以同时拥有多个角色。例如一只基金可以同时是"核心候选 + 卫星阿尔法 + 防守锚 + 低成本"。

角色不是最终结论，而是组合层做权重分配时的输入信号。

### 21.3 风险约束和权重上限

组合层不会让所有基金平均分配权重，而是根据角色和风险标签设置上限。

当前约束定义在 `config/portfolio_constraints.v1.json`，核心规则如下：

**目标桶权重范围：**

| 桶 | 目标权重 | 下限 | 上限 |
|---|---:|---:|---:|
| `core`（核心） | 60% | 45% | 70% |
| `satellite`（卫星） | 30% | 20% | 45% |
| `index_tool`（指数工具） | 10% | 0% | 25% |

**单只基金权重上限：**

| 条件 | 上限 |
|---|---:|
| 默认 | 10% |
| 核心候选 | 15% |
| 卫星阿尔法 | 8% |
| 指数工具 | 20% |

**风险标签额外收紧：**

| 风险标签 | 额外上限 |
|---|---:|
| `drawdown_high`（高回撤） | 3% |
| `volatility_high`（高波动） | 5% |
| `beta_high`（高 Beta） | 4% |
| `holding_concentration_high`（持仓集中） | 6% |
| `industry_concentration_high`（行业集中） | 6% |
| `tracking_error_high`（跟踪误差高） | 8% |

**硬阻断标签：**

如果基金带有以下标签，直接排除出组合草案：

- `benchmark_data_missing`（基准数据缺失）
- `data_insufficient`（数据不足）
- `manual_review_required`（需人工复核）

### 21.4 组合草案

系统会基于角色、评分和约束，自动生成一份组合草案。

草案生成流程：

```text
1. 计算每只基金的综合评分（角色加分 + 风险标签扣分）
2. 按评分排序
3. 排除硬阻断基金
4. 按目标桶比例分配权重
5. 应用单只基金上限和风险上限
6. 超出上限的权重重新分配给同桶其他基金
```

| 指数工具桶基金数 | 约 8 |

### 21.5 人工校准和验收

组合草案不是自动拍板的最终组合。系统提供了人工校准机制：

## 24. 最后总结

这个项目当前已经完成了两个层次的闭环：

**第一层：标签和证据闭环**

```text
不是只告诉你"基金表现怎么样"，
而是告诉你：
  数据够不够，
  应该跟谁比，
  有没有跑赢自己的基准，
  为什么打这个标签，
  哪些结论能展示，
  哪些结论还不能展示。
```

**第二层：组合草案闭环**

```text
在标签之上，进一步告诉你：
  这只基金在组合里应该扮演什么角色，
  能放多少权重，
  哪些风险标签会收紧上限，
  哪些基金被硬阻断排除，
  最终的组合草案长什么样，
  研究员需要签署确认哪些决策。
```

这就是 Phase1 v1 的核心价值。
