## 2. 项目定位

### 2.1 为什么要做

基金研究、产品池管理和后续 FOF 研究里，很多判断过去依赖人工经验、分散表格和临时口径，主要问题是：

- 标签口径不统一：同一个“稳健”“成长”“高波动”在不同分析师之间可能有不同解释。
- 证据链不稳定：很难追溯一个标签是由哪个数据、哪个指标、哪个阈值触发。
- 数据不足时容易误判：缺少 NAV、持仓、经理、费率或股票因子时，系统应该提示观察或复核，而不是强行输出正式结论。
- 难以批量复核：人工可以处理少量基金，但很难稳定处理产品池、观察池和后续扩展池。
- 后续扩展缺底座：如果没有批次、规则版本、证据和复核记录，后面做工作台、回放、审计和 FOF 初筛都会困难。

### 2.2 项目目标

建设一个面向证券资管公司的基金标签计算引擎：

- 输入：基金基础信息、净值、持仓、行业、经理、费率、规模和可选股票因子。
- 输出：数据覆盖率、特征值、标签结果、触发证据、人工复核建议。
- 使用方式：本地批量计算，结果落库，通过 API 和工作台查询。
- 核心原则：规则优先、证据优先、数据不足不强行下正式结论。

### 2.3 项目边界

明确在项目内：

- 基金数据标准化输入。
- 数据覆盖率检查。
- 特征计算。
- 规则标签计算。
- 标签证据生成。
- 标签批次记录。
- 人工复核数据模型。
- API 查询和轻量工作台。

明确不做：

- 不做自动交易。
- 不做实盘委托。
- 不给最终买入、卖出、申购、赎回建议。
- 不用 LLM 自动决定基金准入。
- 不在没有股票因子或足够 NAV 历史时输出强风格/强收益风险结论。
- 不一次性覆盖全市场所有基金类型。

## 3. 当前系统能力

### 3.1 已完成主链路

当前系统主链路如下：

```text
fundData / sample SQLite
  -> 数据覆盖率检查
  -> 特征计算
  -> 标签规则计算
  -> 标签证据生成
  -> 结果落库
  -> API 查询 / 前端工作台 / 人工复核
```

这个链路的价值是：每个标签都不是一句孤立结论，而是一条可以追溯的数据记录，包含指标值、阈值、来源和解释。

## 8. 可直接照着讲的 3 分钟稿

这个项目是一个基金标签计算引擎，服务证券资管投研、产品池管理和后续 FOF 研究。我们不是做基金推荐机器人，也不是做自动交易，而是把本地基金数据加工成可解释、可追溯、可复核的标签结果。

目前主链路已经打通：从 `fundData` 读取基金基础信息、净值、持仓、行业、经理、费率和规模，先做数据覆盖率检查，再计算特征和标签，最后把标签、证据、特征、覆盖率和复核记录落库，通过 API 和前端工作台查看。

真实数据已经试跑。首期只纳入权益相关基金，包括股票型、偏股混合、灵活配置混合和股票指数基金，共 168 只。当前 147 只基础数据齐备，可以输出基础标签；21 只因为缺少必要字段进入人工复核。后端自动化测试 77 条通过，前端构建也通过。

当前最重要的边界有两个。第一，NAV 历史现在太短，不能正式输出 1Y / 3Y 收益风险标签；系统会输出 `return_window_insufficient`，不会强行年化后下结论。第二，真实库目前还没有完整股票因子，所以不能正式输出深度价值、质量成长、红利稳健等高级风格标签；系统会明确提示缺少股票因子。

所以当前项目的价值是：先把基础标签闭环、数据覆盖率、证据链和人工复核机制跑通。下一步不建议盲目扩标签，而是先补齐 1Y NAV 历史、建设股票因子层、完善覆盖率报告和复核工作台。等数据和复核样本稳定后，再开放高级风格标签和更复杂的同类分位、基准相对规则。

## 10. 可能被问到的问题

### Q1：现在这个项目能不能直接用？

可以用于内部试跑、标签口径验证、数据覆盖率检查和复核流程验证。  
但还不建议直接作为正式准入结论，因为 1Y / 3Y NAV 和股票因子还没补齐。

### Q2：为什么不直接用外部 GitHub 项目？

外部项目可以借鉴模块，例如数据接入、风险指标、前端分析页面、Agent 分析思路等；但不适合整仓复用。

原因是我们的目标不是推荐、预测或自动分析，而是证券资管场景下的可审计标签系统，需要：

- 统一数据契约。
- 规则版本管理。
- 标签证据落库。
- 数据不足降级。
- 人工复核。
- 批次回放。

这些通常不是通用基金分析项目的核心。

### Q3：为什么有些标签不输出？

这是项目的质量控制。缺数据时不能强行下结论。

例子：

- 没有股票因子，就不能正式判断深度价值、质量成长、红利稳健。
- NAV 历史不足，就不能正式给 1Y / 3Y 收益风险标签。
- 缺持仓、费率、经理等必要字段，就应该进入 `manual_review_required`。

### Q4：下一步最应该先做什么？

先补数据，再扩标签。

优先级：

1. 补齐首期基金 NAV 历史。
2. 建股票因子层。
3. 建覆盖率报告。
4. 完善复核工作台。
5. 再逐步开放高级风格标签。

### Q5：这个项目后面能服务什么业务场景？

- 产品池准入。
- 观察池管理。
- 基金画像。
- 人工复核队列。
- FOF 初筛。
- 投研标签沉淀。
- 规则回放和审计。

### Q6：为什么首期不覆盖债基、货基、QDII、FOF？

因为这些基金类型的风险来源、收益来源和评价口径不同。

权益基金重点看持仓、行业、权益仓位、风格暴露、波动回撤；债基要看久期、信用、杠杆、票息、回撤来源；货基看流动性和安全性；QDII 看区域、汇率和海外资产；FOF 看底层基金穿透。

如果第一版就全覆盖，规则会变得很泛，不利于形成可信闭环。
