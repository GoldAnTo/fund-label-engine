# 需求文档

## 目标

建设一个基金标签计算引擎，把基金数据转化为可解释、可追溯、可重复计算的标签结果。当前阶段先把“机器计算”闭环做稳：一只基金进来后，系统必须说明哪些标签触发、哪些没有触发、哪些因为数据不足无法计算，以及对应证据和原因。

## 第一版用户

- 投研人员：查看基金画像、收益风险、持仓结构、标签证据。
- 产品池审核人员：判断基金是否进入观察池、是否需要人工复核。
- 数据/系统开发人员：维护数据契约、标签定义、计算批次和结果审计。

## 第一版输入

- 基金基础信息：代码、名称、类型、成立日期、规模、基金公司。
- 净值/收益序列：日频或周频净值、阶段收益。
- 持仓数据：股票持仓、债券持仓、行业配置。
- 基金经理：经理名称、任期、变更情况。
- 费率数据：管理费、托管费、销售服务费。
- 可选股票因子：PB、ROE、股息率、营收增长、利润增长、市值、行业、估值分位。

## 第一版输出

- 数据覆盖率和缺失项。
- 基金特征值，例如最大回撤、波动、夏普、持仓集中度、行业集中度。
- 标签结果，例如 `data_sufficient`、`holding_concentration_high`、`manager_tenure_long`。
- 标签证据，例如指标值、阈值、数据来源、解释文本。
- 标签计算状态，例如 `triggered`、`not_triggered`、`not_computed`。
- 批次摘要，例如标签分布、计算状态分布、未计算原因分布。

## 必须满足的原则

1. 每个标签必须能解释。
2. 每个标签必须保存证据。
3. 数据不足时不强行输出正式标签。
4. 高级风格标签必须依赖股票因子或股票标签。
5. 规则需要可版本化，方便未来回溯历史计算。
6. 人工复核和自动计算需要分开保存；当前阶段只处理自动计算，复核工作台暂缓。

## MVP 验收

- 能对单只基金执行标签计算。
- 能批量计算一组基金。
- 能输出数据覆盖率、标签、证据、计算状态和未计算原因。
- 能落库保存标签定义和计算批次，支持后续审计。
- 能导出批次标签、证据、特征、覆盖率和计算状态。
- 缺少股票因子时，不输出深度价值、质量成长、红利稳健等正式风格标签。
- 测试覆盖覆盖率检查、基础标签、证据生成、高级标签边界和计算状态。

## 当前实现边界

- 数据不完整时，不再直接中止计算；系统会输出 `data_insufficient` 和 `manual_review_required`，同时保留已有证据可以支撑的观察型标签。
- 已有股票因子但高级风格规则未启用时，只输出 `style_pending_rule_definition`，不伪造正式风格标签。
- 股票因子读取按基金持仓报告期做 as-of 对齐，只使用不晚于持仓报告期的最新因子。
- 计算状态单独落 `label_calculation_states`，不污染已触发标签表；这样前端既能看“打了什么标签”，也能看“为什么没打标签”。
- 当前只支持 SQLite 本地运行；复核队列、人工确认和权限流程暂缓，不作为本阶段验收条件。
