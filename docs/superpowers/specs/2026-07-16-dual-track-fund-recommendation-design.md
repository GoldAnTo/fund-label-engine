# 双轨基金推荐与组合输出设计

## 1. 目标

把当前的 CandidateSet（与 Thesis 有关的基金集合）升级为用户可直接理解的推荐结果：

- 每条 Thesis 分别给出主动基金和 ETF/指数基金的推荐榜单。
- 每个榜单默认给出 3 个推荐和 2 个备选。
- 推荐结果再作为组合构建层的唯一可投候选来源，输出默认的混合型最终组合。

本阶段的核心判断是：**主题暴露纯度优先，基金质量和风险用于确认这种暴露是否适合被推荐。**

## 2. 已确认的产品决策

| 决策 | 结论 |
|---|---|
| 推荐与组合的关系 | 先选基金，再配组合；组合不能直接从全部 CandidateSet 配置。 |
| 产品分类 | 主动基金与 ETF/指数基金分开推荐，不做跨类别混排。 |
| 核心排序原则 | 主题暴露纯度优先。 |
| 输出数量 | 每类默认 3 个推荐、2 个备选。 |
| 默认组合偏好 | 主动基金与 ETF/指数基金混合。 |
| 决策审批 | 不在本阶段范围内。 |

## 3. 边界与术语

```text
CandidateSet：与 Thesis 相关、可被评价的基金全集，不等于推荐。
RecommendationRun：某个 CandidateSet 在一个快照和推荐政策下的评价结果。
RecommendationResult：一只基金在一个 RecommendationRun 中的分类、得分拆解和解释。
Recommended universe：每个榜单的推荐与备选基金；是组合构建的唯一输入。
PortfolioProposal：在推荐集合内去重、约束、配权后形成的组合建议。
```

不包含：收益预测、未来排名保证、买卖指令、自动批准或状态流转改造。

## 4. 总体数据流

```text
ResearchInput -> InvestmentThesis -> CandidateSet
                                  -> RecommendationRun
                                  -> ActiveFund榜单 / ETFIndex榜单
                                  -> 推荐池
                                  -> PortfolioProposal
                                  -> 风险二次裁决与调权后指标
```

CandidateSet 保留全量映射证据，RecommendationRun 只负责可重复的选基评价，PortfolioProposal 只负责在已推荐资产中配权。三者必须携带同一 `thesis_id`、`candidate_set_id`、`data_snapshot_id` 和策略版本引用。

## 5. 推荐评价

### 5.1 分类

每只 CandidateSet 基金必须先归入下列一种类别：

- `active_fund`：主动管理基金。
- `etf_or_index`：ETF 或指数基金。
- `unsupported`：无法可靠识别产品类别；不能进入推荐，保留理由。

两类产品独立应用门槛、独立排序和独立产生 Top 3 / 备选 2。不得将主动基金的经理能力与 ETF 的跟踪质量放进同一总分横向比较。

### 5.2 硬门槛

任一条件不满足时标记 `excluded`，不能进入推荐或备选：

- 没有可识别的主题暴露证据，或主题暴露低于推荐政策的最低值。
- 持仓数据缺失、过期，或披露覆盖率不足以判断主题暴露。
- 产品类别无法识别。
- 现有策略政策标记为硬风险或硬估值冲突。

信息缺失必须输出 `data_insufficient`，不能被解释为质量好或风险低。

### 5.3 可解释评分

通过门槛后，按 100 分计算，评分明细必须和原始证据一并保存：

| 维度 | 权重 | 含义 |
|---|---:|---|
| 主题暴露纯度 | 55 | 优先使用真实目标持仓权重；已披露持仓内的匹配比例只能辅助解释。 |
| 产业链/Thesis 契合 | 15 | 命中因果链、用户股票关键词和受益逻辑的质量。 |
| 风险收益与回撤 | 15 | 同类产品内的风险调整表现、波动和最大回撤；不以短期收益单独决定推荐。 |
| 基金质量 | 15 | 主动基金看经理稳定性、规模、费率与持仓稳定性；ETF/指数基金看指数主题纯度、费率、规模流动性与跟踪质量。 |

评分不足以替代门槛。高分基金如果数据不足或触发硬门槛，仍不可推荐。

### 5.4 分档和排序

- `recommended`：各产品类别内，满足门槛且排名前 3。
- `alternative`：同类后续 2 个，或前 3 不足时所有仍合格的基金。
- `watch`：主题相关但估值、数据时效或软风险需要继续观察。
- `excluded`：命中硬门槛。
- `data_insufficient`：无法可靠评价。

同档排序先按总分降序，再按主题暴露纯度、数据新鲜度和基金代码稳定排序，保证相同输入可复现。

## 6. 面向用户的输出

每个榜单必须显示：

- 推荐/备选/观察/排除标签与排名。
- 主题暴露、产业链契合、风险收益、基金质量四项得分及总分。
- `推荐理由`：三条以内、来自结构化证据。
- `未选原因`：适用于备选、观察、排除和数据不足项目。
- 证据时间与数据快照，避免把过期持仓当成当前结论。

页面首先展示“主题基金推荐”，主动基金榜单与 ETF/指数基金榜单并列或分 Tab；其后展示“最终组合建议”。用户应能清楚区分“基金本身为何被选中”与“它在组合中为何只配这个权重”。

## 7. 组合构建

PortfolioProposal 的输入只能是每个 Thesis 的 `recommended` 与 `alternative` 基金。默认策略为混合：主动基金和 ETF/指数基金都可入选。

组合层职责：

- 在多个 Thesis 的推荐池中去除同代码和高度重叠基金。
- 在主动基金和 ETF/指数基金之间分配表达方式，避免同一主题双重放大。
- 基于用户 `risk_tolerance`、行业暴露、持仓重叠、回撤和波动率约束生成权重。
- 复用现有风险二次裁决；任何强制降权后都重算组合指标与裁决结果。

组合层不得以低主题暴露基金填充风险预算。推荐池无法形成合格组合时，返回结构化的“推荐不足/需要补充防守资产”说明，而不是伪造完整配置。

## 8. 持久化与接口

新增 RecommendationRun / RecommendationResult 治理对象，记录推荐政策、运行时间、输入快照和结果。它们与现有 CandidatePriorityRun 分工不同：

- CandidatePriorityRun：研究优先级，回答“先研究什么”。
- RecommendationRun：用户推荐，回答“在本 Thesis 下优先选择什么”。

前端/接口输出需要同时保留：`thesis_id`、`candidate_set_id`、`recommendation_run_id`、`data_snapshot_id`、`strategy_policy_id`。PortfolioProposal 额外引用其来源 `recommendation_run_id` 列表，以支持回放和解释。

## 9. 失败处理

- 没有符合门槛的主动基金或 ETF/指数基金是正常结果，返回空榜单和明确原因。
- 数据缺失只影响对应基金或榜单，不阻断 Thesis、CandidateSet 或另一类别的推荐。
- 推荐运行持久化失败时，接口不得把临时结果伪装成可审计结果；应显式标记 `persistence_status=degraded`。
- 组合输入不足时返回未完成组合，不执行虚假的权重归一化。

## 10. 验收标准

1. 同一 CandidateSet 中的主动基金和 ETF/指数基金分别排序，且互不影响名次。
2. 主题暴露纯度在两类榜单中均为最大权重，并可在单基金详情中追溯。
3. 每类默认最多 3 个推荐和 2 个备选，缺少合格基金时如实返回较少数量。
4. 数据不足、硬风险或硬估值冲突的基金不会被推荐。
5. 每个推荐、备选与未选基金均有结构化理由和数据快照引用。
6. 组合只从推荐/备选池取基金，并在风险执行后展示调权动作与更新后的指标。
7. 相同 Thesis、CandidateSet、快照和政策版本可重放出相同的推荐结果。
