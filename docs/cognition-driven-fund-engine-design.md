# 认知驱动的基金配置引擎设计方案

> 基于会议修改.md 的核心洞察，重新定义项目方向：从"图书馆管理员"式的标签分类，升级为"投资者认知 → 基金配置"的转化引擎。
>
> 创建时间：2026-07-08

---

## 一、为什么要改：会议暴露的核心问题

### 1.1 当前项目的困境

现有标签引擎已经做得很扎实：142 只基金、50+ 标签、证据链、基准解析、分类分组、组合草案。但它陷入了会议中描述的"数据迷宫"：

```
做了很多 → 但不知道出口在哪里
标签很全 → 但回答不了"我该买什么"
排名很细 → 但没有服务于"我相信什么"
```

用会议中的比喻：**像图书馆管理员，把书分好了类（文学、历史、哲学…），但回答不了"哪本哲学书最好"。**

### 1.2 三个根本性问题

| 问题 | 会议原文提炼 | 现状 |
|---|---|---|
| 基金是壳子 | "基金是一个装满糖果的盒子，你要研究的是糖果不是盒子" | 只研究盒子（基金层），没穿透到糖果（底层资产逻辑） |
| 线性外推不成立 | "过去涨的多的未来涨的多很难，后视镜数据看将来会出问题" | 标签全部基于历史数据，本质是线性外推 |
| 投资本质是"相信什么" | "我相信什么我投什么" | 没有"认知"这个输入维度，只有数据输出 |

### 1.3 现有组合草案的方向性问题

现有组合草案是**自下而上、数据驱动**的：

```
基金标签 → 角色推导（核心/卫星/防守） → 评分排序 → 权重分配
```

但会议要求的是**自上而下、认知驱动**的：

```
我相信什么 → 验证/纠正 → 找匹配的基金 → 看估值 → 构建组合
```

两者方向完全不同。现有方案是"给你一堆基金，帮你分类排序"，会议要求的是"给我一个认知，帮我找到匹配的基金"。

## 二、核心理念转变

### 2.1 从"标签分类"到"认知转化"

```
旧方向：基金数据 → 标签计算 → 分类分组 → 排序推荐
                   ↓ 陷入数据迷宫

新方向：投资者认知 → 认知验证 → 资产映射 → 估值检查 → 组合配置
                   ↓ 出口明确：给认知一个匹配的配置方案
```

### 2.2 保留什么，改变什么

| 维度 | 保留（作为底座） | 改变（新增方向） |
|---|---|---|
| 标签引擎 | 标签计算、证据链、基准解析 | 不再是产品出口，而是认知验证的证据层 |
| 基金数据 | NAV、持仓、因子、费率 | 新增底层资产穿透和估值判断 |
| 组合层 | 风险约束、权重上限 | 从"数据排序分配"改为"认知匹配分配" |
| 前端 | 基金诊断、竞品横评 | 新增认知输入 → 配置输出的交互入口 |

### 2.3 会议中的三个核心案例

新方案必须能回答这三个问题：

**案例 1：我喜欢塞尔达传说，想买任天堂相关的基金**
```
输入认知：看好任天堂（塞尔达系列）
系统输出：
  → 穿透持仓，找到任天堂占比最高的基金
  → 但同时显示：任天堂当前 PE XX 倍，历史分位 XX%
  → 估值判断：偏贵/合理/便宜
  → 配置建议：如果坚信，可以配置 X%；如果只是喜欢产品，不建议买入
```

**案例 2：我相信 AI 是巨大生产力变革**
```
输入认知：看好 AI 产业链
系统输出：
  → AI 产业链拆解：基础设施（台积电/三星/美光）→ 应用层（MiniMax/OpenAI）
  → 找到各环节占比最高的基金
  → 检查估值：寒武纪 300 倍 PE vs 某公司 30 倍 PE
  → 警示：哪些已经 price in 了太多未来增长
  → 给出配置清单 + 比例建议
```

**案例 3：我相信创新药被错杀**
```
输入认知：看好创新药
系统输出：
  → 创新药定义：不是所有医药都是创新药
  → 找到创新药占比高的基金（不是简单看"医药主题"标签）
  → 检查估值：创新药板块当前 PE/PB 分位
  → 检查基本面：BD 交易数据、海外授权进展
  → 配置建议
```

## 三、新架构设计

### 3.1 总体架构

```
┌─────────────────────────────────────────────────────────────┐
│                    认知驱动的基金配置引擎                       │
│                                                             │
│  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐   │
│  │  认知输入层  │ →  │  认知验证层   │ →  │  资产映射层  │   │
│  │             │    │              │    │              │   │
│  │ "我相信什么" │    │ 数据验证 +   │    │ 认知 → 资产  │   │
│  │  主题/赛道/  │    │ 估值检查 +   │    │ → 基金穿透   │   │
│  │  个股/风格   │    │ 基本面检查   │    │ → 持仓匹配   │   │
│  └─────────────┘    └──────────────┘    └──────────────┘   │
│                                                   ↓         │
│  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐   │
│  │  组合输出层  │ ←  │  估值门禁层   │ ←  │  组合构建层  │   │
│  │             │    │              │    │              │   │
│  │ 配置清单 +   │    │ 贵不贵？     │    │ 认知匹配度   │   │
│  │ 比例建议 +   │    │ 分位多少？   │    │ 评分 + 权重  │   │
│  │ 风险提示     │    │ price in 了  │    │              │   │
│  │             │    │ 多少增长？    │    │              │   │
│  └─────────────┘    └──────────────┘    └──────────────┘   │
│                                                             │
│  底座：现有标签引擎（证据层）+ 基金数据 + 股票因子             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 六层架构详解

#### 第 1 层：认知输入层

投资者输入"我相信什么"。认知有四种类型：

| 认知类型 | 举例 | 输入格式 |
|---|---|---|
| 主题认知 | "我相信 AI 是巨大生产力变革" | 主题关键词 + 产业链环节 |
| 赛道认知 | "我相信创新药被错杀" | 行业 + 细分赛道 |
| 个股认知 | "我喜欢任天堂/塞尔达" | 股票名称/代码 |
| 风格认知 | "我相信龙头集中" | 风格描述 |

**认知结构化**：把自然语言认知拆解成机器可处理的结构：

```python
CognitionInput = {
    "belief": "我相信 AI 是巨大生产力变革",
    "type": "theme",
    "theme": "AI",
    "chain_links": ["基础设施", "应用层"],  # 产业链环节
    "time_horizon": "long",  # 短期/长期
    "conviction": "high",  # 信心强度
    "risk_tolerance": "moderate",  # 风险偏好
}
```

#### 第 2 层：认知验证层

把认知作为假设，用数据验证或纠正。不是替代投资者判断，而是提供证据。

| 验证维度 | 做什么 | 用什么数据 |
|---|---|---|
| 基本面验证 | 这个认知有没有基本面支撑？ | 行业增速、企业盈利、BD 交易等 |
| 估值检查 | 这个认知被 price in 了多少？ | PE/PB 分位、市值/营收比 |
| 市场情绪 | 市场对这个认知什么态度？ | 资金流向、板块涨幅、拥挤度 |
| 反面证据 | 有什么数据不支持这个认知？ | 亏损数据、竞争格局变化 |

**关键原则**：会议说"有些信息左右矛盾，最终靠相信"。所以验证层不是给结论，而是给证据，让投资者自己判断。

```python
CognitionValidation = {
    "belief": "我相信 AI 是巨大生产力变革",
    "supporting_evidence": [
        "美国名义 GDP 增速达到 50 年高位",
        "AI 基础设施收入确认中",
    ],
    "opposing_evidence": [
        "终端应用变现仍在亏损",
        "部分公司估值已 price in 未来 5 年增长",
    ],
    "valuation_check": {
        "ai_infra_avg_pe": 45,
        "ai_infra_pe_percentile": 85,  # 历史分位
        "assessment": "偏贵，已 price in 较多增长预期",
    },
    "verdict": "partially_validated",  # validated / partially_validated / unvalidated
}
```

#### 第 3 层：资产映射层

把认知映射到具体的底层资产，再从底层资产穿透到基金。

这是会议核心：**"基金是壳子，本质是股票组合"**。

```
认知："我相信 AI"
    ↓
资产映射：
    AI 基础设施 → 台积电、三星、美光、海力士
    AI 应用层 → MiniMax、OpenAI（未上市）
    AI 芯片 → 寒武纪、英伟达
    ↓
基金穿透：
    扫描 142 只基金的持仓，找到上述股票占比最高的基金
    输出：基金 A（台积电占比 8.5%）、基金 B（三星占比 6.2%）...
```

**资产定义库**：需要建立一个"认知 → 资产"的映射库

```python
ThemeAssetMap = {
    "AI": {
        "infrastructure": {
            "stocks": ["台积电", "三星电子", "美光科技", "SK海力士"],
            "sectors": ["半导体", "存储芯片"],
        },
        "application": {
            "stocks": ["寒武纪", "海光信息"],
            "sectors": ["AI 软件", "大模型"],
        },
    },
    "innovation_drug": {
        "stocks": ["百济神州", "信达生物", "恒瑞医药"],
        "sectors": ["创新药", "CXO"],
    },
    # ... 可扩展
}
```

#### 第 4 层：估值门禁层

会议反复强调：**"热爱产品 ≠ 买股票，要看估值"**。

这一层对每个匹配的基金/资产做估值检查：

| 检查项 | 做什么 | 输出 |
|---|---|---|
| 估值分位 | 加权 PE/PB 在历史什么位置 | P30 以下便宜 / P70 以上偏贵 |
| 预期 price in | 当前估值隐含了多少年增长 | 隐含 3 年 / 5 年 / 10 年 |
| 安全边际 | 当前价格 vs 合理估值 | 有安全边际 / 已透支 |
| 估值趋势 | 估值在扩张还是收缩 | 扩张（市场看好）/ 收缩（市场看空） |

**估值判断逻辑**（以会议中三星的例子说明）：

```
三星 PE 30 倍，过去一年利润超过去 40 年总和
如果未来 40 年维持这个利润 → PE 变成 1200 倍 → 不合理
所以当前估值可能已 price in 过多增长 → 偏贵
```

#### 第 5 层：组合构建层

基于认知匹配度 + 估值判断，构建组合。

```
认知匹配度评分：
    基金 A 持仓中 AI 相关占比 35% → 匹配度高
    基金 B 持仓中 AI 相关占比 12% → 匹配度中
    基金 C 持仓中 AI 相关占比 3% → 匹配度低

估值调整：
    基金 A 加权 PE 85 倍（P90）→ 偏贵，降低权重上限
    基金 B 加权 PE 30 倍（P50）→ 合理，正常权重

组合输出：
    基金 A：认知匹配度高但估值贵 → 配置 5%（受估值上限约束）
    基金 B：认知匹配度中且估值合理 → 配置 8%
    + 防守仓位（红利低波）→ 配置 15%（对冲认知错误的风险）
```

**新增"认知对冲"机制**：会议中导师提到"为了防止高位风险，选一部分红利低波作为防守"。系统应该自动建议对冲仓位。

#### 第 6 层：组合输出层

最终给投资者一份"认知匹配的配置方案"：

```
┌──────────────────────────────────────────────┐
│  你的认知：我相信 AI 是巨大生产力变革           │
│                                              │
│  认知验证：                                   │
│  ✅ 基本面：AI 基础设施收入高速增长             │
│  ⚠️ 估值：AI 板块整体偏贵（PE P85）            │
│  ⚠️ 风险：终端变现尚未证明                     │
│                                              │
│  配置方案：                                   │
│  ┌────────────┬────────┬──────┬──────────┐  │
│  │ 基金       │ 匹配度 │ 估值 │ 建议权重  │  │
│  ├────────────┼────────┼──────┼──────────┤  │
│  │ 基金A      │ 35%    │ 偏贵 │ 5%       │  │
│  │ 基金B      │ 18%    │ 合理 │ 8%       │  │
│  │ 基金C(防守)│ —      │ 低估 │ 7%       │  │
│  └────────────┴────────┴──────┴──────────┘  │
│                                              │
│  风险提示：                                   │
│  · AI 板块估值处于历史 85% 分位，回撤风险大    │
│  · 终端应用仍在亏损，若变现不及预期可能杀估值   │
│  · 建议控制 AI 主题总仓位不超过 20%            │
│                                              │
│  持仓穿透：                                   │
│  · 台积电 6.8%（基金A+B合计）                 │
│  · 三星 4.2%                                 │
│  · 寒武纪 2.1%                               │
│  · 组合加权 PE 42 倍（全市场 P70）            │
└──────────────────────────────────────────────┘
```

**关键原则**：不推翻现有标签引擎，而是在其上层加一个"认知驱动层"。标签引擎从"产品出口"变为"证据底座"。

## 五、数据需求与缺口

### 5.1 现有数据可直接复用

| 数据 | 用途 | 现状 |

|---|---|---|---|

**局限**：142 只都是 A 股权益基金，但会议案例涉及台积电、三星、任天堂等港股/美股。

**应对**：
- 短期：先支持 A 股主题（AI 芯片、创新药、消费等），用 142 只基金验证流程
- 中期：扩展沪港通基金的持仓数据
- 长期：支持全球资产配置

## 九、双轨基金推荐系统

### 9.1 四阶段定位

认知引擎从"投资者认知"到"可执行组合"经过四个阶段，每个阶段有独立的运行记录和职责边界：

| 阶段 | 对象 | 职责 | 回答的问题 |
|---|---|---|---|
| CandidateSet | 候选集合 | 扫描全量基金，冻结证据快照 | "跟这个主题相关的基金有哪些？" |
| CandidatePriorityRun | 优先级评价 | 按数据完备度排序 | "应该先研究哪些基金？" |
| **RecommendationRun** | **推荐评价** | **主题优先、主动/ETF 分开排序** | **"推荐买哪些基金？"** |
| PortfolioProposal | 组合提案 | 只消费推荐池，执行风险裁决 | "组合权重怎么分配？" |

CandidateSet 是候选池，PriorityRun 只回答研究顺序（不是买入建议），RecommendationRun 才是选基推荐，PortfolioProposal 只能消费推荐和备选结果。

### 9.2 双轨排序机制

RecommendationRun 对主动基金和 ETF/指数基金分别排序，两条赛道独立产出榜单：

```
CandidateSet（全量候选，含冻结证据）
    │
    ├─ 主动基金赛道 ──→ 类内排序 ──→ recommended / alternative / watch
    │
    └─ ETF·指数赛道 ──→ 类内排序 ──→ recommended / alternative / watch
```

产品分类依据 `fund_profiles.fund_type` 元数据（不靠基金名称猜测）。四维评分权重：

| 维度 | 权重 | 含义 |
|---|---|---|
| theme_exposure | 0.55 | 主题暴露纯度（持仓匹配度） |
| thesis_alignment | 0.15 | 投资假设匹配度 |
| risk_return | 0.15 | 风险收益特征 |
| fund_quality | 0.15 | 基金质量（主动看经理/因子/趋势，ETF 看纯度/因子/经理/估值） |

档位定义：`recommended`（推荐）、`alternative`（备选）、`watch`（观察）、`excluded`（排除）、`data_insufficient`（数据不足）。每条赛道按 `active_fund_limit` / `etf_or_index_limit` 截取推荐数量。

### 9.3 组合只消费推荐池

PortfolioProposal 的输入域被严格限定为推荐池（`recommended` + `alternative`），不再回退全量 CandidateSet：

```
RecommendationRun
    │
    └─ recommended_universe（推荐 + 备选）
            │
            └─ PortfolioProposal
                    ├─ 去重 / 重叠度裁决
                    ├─ 行业集中度裁决
                    ├─ 波动率 / 回撤裁决
                    └─ 调权后重算指标
```

当推荐池为空时，组合状态为 `insufficient_recommendations`（不生成假组合）。组合输出固定包含 `selection_source='recommended_universe'` 和 `recommendation_run_ids`，确保可审计。

### 9.4 不可变记录与幂等

RecommendationRun 和 Result 表设有 `UPDATE`/`DELETE` 触发器（报错含 `immutable`），保证推荐结果一经写入不可篡改。幂等键为 `(candidate_set_id, strategy_policy_id, strategy_policy_version, data_snapshot_id, recommendation_method_version)`，重复请求返回已存在的 run_id（409 Conflict）。

## 十、核心设计原则

1. **认知优先，数据服务于认知**：不是"给你数据你自己看"，而是"给我认知我帮你找匹配"。
2. **穿透壳子，看到底层资产**：基金是壳子，核心是它装了什么股票。
3. **估值是硬约束**：热爱产品 ≠ 买股票，估值太贵时必须提示风险。
4. **不替代判断，只提供证据**：会议说"信息矛盾时靠相信"，系统给证据不给结论。
5. **保留底座，不推翻重来**：现有标签引擎是资产，新方案在其上层构建。

## 十一、总结

这个方案的核心转变是：

```
从：基金数据 → 标签 → 分类 → 排序 → "这是分类结果"
到：投资者认知 → 验证 → 穿透 → 估值 → "这是匹配你认知的配置方案"
```

会议的"出口"就在这里：**标签引擎的出口不是更全的标签，而是把投资者的认知转化为一个可执行的配置方案。**
